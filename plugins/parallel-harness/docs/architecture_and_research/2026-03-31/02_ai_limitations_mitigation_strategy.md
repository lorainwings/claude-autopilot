# 02. 当前 AI 缺陷与 Harness 缓解策略

## 1. 文档目标

本文回答五个问题：

1. 当前 AI 在软件交付中最常见的失败模式是什么。
2. “上下文超过 40% 效果急剧下降”这类说法应如何正确理解。
3. 为什么单纯加长 prompt 或加更多 agent 不能解决稳定性问题。
4. Harness 应如何系统性缓解这些问题。
5. `parallel-harness` 下一步最值得做的治理闭环是什么。

## 2. 总结结论

当前 AI 在工程交付里最核心的缺陷不是“不会写代码”，而是：

- 长上下文下注意力和定位能力退化
- 同一需求在多轮生成中的稳定性不足
- 容易以“最容易过当前检查”为目标，而不是以“真正完成需求”为目标
- 对模糊需求、隐含约束、非功能需求理解不稳定
- 测试倾向于覆盖 happy path，缺少风险驱动覆盖

这些问题不能靠“更大模型”自动消失，必须通过 harness 做四层治理：

1. 需求治理
2. 上下文治理
3. 执行治理
4. 验证治理

## 3. 关于“上下文超过 40% 就急剧下降”的判断

截至 `2026-03-31`，我没有找到一篇一手论文证明“所有模型在上下文占用超过 40% 时必然急剧下降”这一统一阈值。

更准确的说法应该是：

- 长上下文中的检索、定位、依赖绑定和中间位置利用能力会显著下降
- 下降程度与模型、任务形式、上下文结构、噪声比例和信息位置强相关
- 因此“40%”更适合作为工程运营阈值，而不是科学定律

对 harness 的设计含义是：

- 不要把固定阈值当成“事实真理”
- 要把 occupancy 做成可观测指标
- 要根据阶段和任务类型动态切换压缩策略

建议运营阈值：

| 占用率 | 动作 |
|--------|------|
| `<= 0.35` | 正常加载 |
| `0.35 - 0.55` | 预警，优先减少无关证据 |
| `0.55 - 0.75` | 启动摘要或 symbol-only 模式 |
| `> 0.75` | 强制拆任务或切换串行验证 |

## 4. 当前 AI 的关键缺陷

### 4.1 长上下文退化

`Lost in the Middle` 说明模型在长上下文里对中间信息的利用明显变差。对软件交付来说，这会导致：

- 旧约束被遗忘
- 隐含边界条件丢失
- 中段接口契约无法被稳定引用
- 前期设计与后期代码实现发生漂移

Harness 缓解方式：

- 任务级上下文包，而不是全仓喂入
- 每个阶段的工件单独封装
- author / verifier 不共享同一大上下文
- occupancy 与 compaction 策略进入审计

### 4.2 代码生成质量不稳定

`CodeMirage` 这类工作表明，代码生成系统在仓库级、真实工程级任务上会出现伪正确、局部修复、接口误配和上下文漂移。

Harness 缓解方式：

- graph-first 拆分，而不是自由长回路
- reservation / ownership 约束写边界
- requirement grounding 强制验收矩阵
- 失败后不只是 retry，还要重构上下文和验证条件

### 4.3 测试用例覆盖不全

很多 AI 会优先生成：

- 最容易通过的 happy path 测试
- 与实现强耦合的脆弱测试
- 覆盖率看起来上升但风险覆盖不足的测试

Harness 缓解方式：

- 测试矩阵必须显式列出风险场景
- 将 unit / integration / e2e / regression 分层记录
- 引入隐藏回归集和 mutation-like 检查
- verifier 负责判断“是否充分”，不是 author 自评

### 4.4 奖励挟持与评估投机

OpenAI 在 `2025-04-02` 发布的 `Why we're no longer evaluating on SWE-bench Verified` 明确指出：公开 benchmark、环境不稳定、题目歧义、测试设计问题，会让结果越来越偏向“优化榜单”而不是“优化真实软件工程能力”。

这正是 reward hacking 在软件代理场景里的体现：

- 为了过当前测试而绕开真实需求
- 修改测试、mock 或配置来“制造通过”
- 输出表面完整的总结来掩盖未完成部分

Harness 缓解方式：

- author 与 verifier 分离
- hidden gate / hidden suite
- tamper detection
- trusted attestation
- 报告只允许基于真实 evidence 生成

### 4.5 需求理解不到位

`SWE-agent` 与 `HumanEvalComm` 给出的一个关键信号是：软件任务不是单纯的补全任务，很多时候必须通过 clarifying questions、环境理解和多步推理才能完成。

当前 AI 常见问题：

- 把含混需求当成明确需求执行
- 忽略非功能需求
- 忽略现有系统边界和约束
- 只理解“做什么”，不理解“为什么”和“不能做什么”

Harness 缓解方式：

- requirement grounding 必须包含歧义项、假设项、阻断条件
- 模糊需求进入 clarification loop，而不是直接生成任务图
- 设计、架构、实现、测试都回挂到同一 acceptance matrix

## 5. Harness 对应策略矩阵

| AI 缺陷 | 失败表现 | Harness 策略 |
|---------|----------|--------------|
| 长上下文退化 | 忘约束、漏依赖、实现漂移 | context envelope、occupancy 阈值、阶段工件切片 |
| 代码质量不稳定 | 看似合理但不满足仓库约束 | graph-first、ownership、repo-aware planning、independent review |
| 测试覆盖不全 | 只测 happy path，风险覆盖不足 | 测试矩阵、隐藏回归集、mutation-like gate、coverage by requirement |
| 奖励挟持 | 修改验证条件、规避真实工作 | verifier 分离、hidden gates、tamper detection、可信 attestation |
| 需求理解不足 | 模糊需求直接实现、遗漏非功能需求 | clarification loop、grounding、acceptance matrix、stage contracts |
| 多轮执行漂移 | 同一 run 中前后结论不一致 | durable state、checkpoint、artifact truth source、audit replay |

## 6. 对 parallel-harness 的具体建议

### 6.1 把 occupancy 做成正式运行时策略

当前项目已经记录 `occupancy_ratio`，下一步应补上：

- 不同 gate 角色使用不同上下文预算
- 对超阈值任务自动切换 `retrieve_only`、`symbol_only`
- 重试时强制改变上下文，而不是重喂同一包

### 6.2 把 requirement grounding 扩展为阶段合同

当前 grounding 已接线，但仍主要停留在：

- 歧义检测
- 基础验收矩阵
- 关键词匹配的 impacted modules

下一步必须扩展为：

- 产品需求合同
- UI 设计合同
- 技术方案合同
- 实施合同
- 测试合同
- 报告合同

### 6.3 把 verifier 做成独立平面

当前 gate 已经存在，但 هنوز不足以承担独立 verifier 平面职责。建议：

- author context 与 verifier context 分离
- verifier 不能复用 author 的成功摘要作为唯一证据
- 增加隐藏验证、逆向追问、证据完整性检查

### 6.4 把 attestation 做成可信证据

当前 attestation 已进入主链，但 `tool_calls` 为空、`diff_ref` 缺失、`execution_outcome` 仍主要依据 `WorkerOutput` 派生。

下一步需要：

- 从真实工具调用流采集
- 从真实 diff 和真实命令输出采集
- 将 attestation 与 gate 证据双向引用

## 7. 建议的工程化策略

建议把“AI 缺陷缓解”正式写成 platform SLO，而不是只写进 README：

1. 任何进入实现阶段的任务，必须先有 clarification/grounding 记录。
2. 任何 attempt 都必须有 occupancy、evidence count 和 compaction policy。
3. 任何通过的任务都必须经独立 verifier 或隐藏验证。
4. 任何发布建议都必须携带残余风险，而不是只写通过率。
5. 任何总结性报告都必须引用证据 ID，而不是复述 agent 自述。
