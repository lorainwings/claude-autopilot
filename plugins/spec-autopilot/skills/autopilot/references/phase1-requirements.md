# Phase 1: 需求理解与多轮决策 — 详细流程

> 本文件由 autopilot SKILL.md 引用，执行 Phase 1 时按需读取。

**核心原则**: 绝不假设，始终列出选项由用户决策。

---

## 1.1 获取需求来源

- `$ARGUMENTS` 为文件路径 → 读取文件内容
- `$ARGUMENTS` 为文本 → 直接作为需求描述
- `$ARGUMENTS` 为空 → AskUserQuestion 要求输入

获取到的原始需求文本记为 `RAW_REQUIREMENT`。

## 1.2 项目上下文扫描（Auto-Scan）

> 当 `config.phases.requirements.auto_scan.enabled`（默认 true）时执行。

**目的**：为后续讨论提供项目级持久上下文，避免每次需求讨论从零开始。

### 持久化上下文检查（v3.0 新增，优先于全量扫描）

在执行扫描前，先检查项目级持久化上下文：

```
persistent_dir = "openspec/.autopilot-context/"

IF persistent_dir exists AND project-context.md exists:
    mtime = file modification time of project-context.md
    IF (now - mtime) < 7 days:
        # 新鲜 → 跳过全量扫描，直接读取
        READ persistent_dir/project-context.md → 注入上下文
        READ persistent_dir/existing-patterns.md → 注入模式
        READ persistent_dir/tech-constraints.md → 注入约束
        # 仅对本次需求相关代码做增量扫描（Grep 需求关键词相关文件）
        SKIP 全量 Steering Documents 生成
    ELSE:
        # 过期 → 增量更新
        READ existing files → 作为基线
        RUN 全量扫描 → 与基线 diff
        UPDATE persistent files with delta
ELSE:
    # 首次 → 全量扫描（当前行为）
    RUN 全量扫描
    WRITE results to persistent_dir (new)
    COPY to per-change context/
```

> **幂等性**：持久化文件更新采用读取 → 修改 → 完整写回策略，非追加写入。

### 扫描范围

使用 Glob + Grep + Read 在主线程中扫描以下信息：

| 扫描项 | 方法 | 产出 |
|--------|------|------|
| 技术栈 | Glob: `**/build.gradle*`, `**/pom.xml`, `**/package.json`, `**/go.mod`, `**/Cargo.toml`, `**/requirements.txt` | 语言 + 框架 + 版本 |
| 目录结构 | Glob: `**/src/**`, 结合 `.gitignore` 排除 | 模块布局树（2 层深度） |
| 现有模式 | Grep: 路由定义、API 端点、数据库模型、组件注册 | 已有代码模式清单 |
| 依赖清单 | Read: `package.json`, `build.gradle`, `requirements.txt` 的依赖节 | 关键依赖 + 版本 |
| 项目约束 | Read: `CLAUDE.md`, `.claude/rules/*.md`（前 50 行） | 编码约束清单 |
| 测试基础设施 | Glob: `**/test*/**`, `**/spec*/**`, `**/*.test.*`, `**/*.spec.*` | 现有测试框架 + 覆盖范围 |

### 产出 Steering Documents

扫描结果写入 `openspec/changes/<name>/context/` 目录下的 3 个 Steering 文件：

**`project-context.md`**（项目上下文）：
```markdown
# Project Context — Auto-generated by Autopilot Phase 1

## Tech Stack
- Backend: {language} {version} + {framework} {version}
- Frontend: {framework} {version} + {bundler}
- Database: {type}

## Directory Layout
{2-level module tree}

## Key Dependencies
{top 10 dependencies with versions}

## Coding Constraints (from CLAUDE.md / rules)
{extracted constraints list}

## Existing Test Infrastructure
{test framework, test count, coverage baseline}
```

### 历史知识注入（v2.4.0 新增）

Auto-Scan 完成后，检查 `openspec/.autopilot-knowledge.json` 是否存在：

1. **存在** → 读取 entries 数组，按 tags 与 RAW_REQUIREMENT 关键词匹配
2. 取 top-5 相关度最高的条目
3. 写入 `project-context.md` 的「Historical Knowledge」章节：
   ```markdown
   ## Historical Knowledge (from previous autopilot sessions)

   - [pitfall] {summary}: {detail} (from change: {change_name})
   - [decision] {summary}: {detail} (from change: {change_name})
   - [pattern] {summary}: {detail} (reused {reuse_count} times)
   ```
4. 匹配到的 entries 的 `reuse_count += 1`（延迟写回到 knowledge.json）
5. **不存在** → 跳过（首次运行无历史数据）

> 详见：`references/knowledge-accumulation.md`

**`existing-patterns.md`**（现有代码模式）：
```markdown
# Existing Patterns — Auto-generated by Autopilot Phase 1

## API Patterns
{route definitions, naming conventions, auth middleware}

## Data Models
{entity list with key fields}

## Component Patterns (Frontend)
{component naming, state management, routing}

## Error Handling
{error handling approach, common patterns}
```

**`tech-constraints.md`**（技术约束）：
```markdown
# Technical Constraints — Auto-generated by Autopilot Phase 1

## Hard Constraints (from project rules)
{extracted from CLAUDE.md and .claude/rules/}

## Dependency Constraints
{version pinning, peer dependency requirements}

## Infrastructure Constraints
{deployment target, environment requirements}
```

> **幂等性**：如果 Steering 文件已存在（崩溃恢复场景），跳过扫描，直接读取已有文件。

## 1.3 技术调研（Research Agent）

> 当 `config.phases.requirements.research.enabled`（默认 true）时执行。

**目的**：在用户讨论之前，基于事实而非猜测进行需求分析。

### 调度方式

调用 Task(subagent_type = config.phases.requirements.research.agent，默认 "Explore") 进行技术调研。

此 Task **不含** `autopilot-phase` 标记（设计预期，不受 Hook 门禁）。

### Research Agent Prompt 模板

```
你是 autopilot Phase 1 的技术调研 Agent。

## 需求概述
{RAW_REQUIREMENT}

## 项目上下文
读取以下文件获取项目背景：
- openspec/changes/<name>/context/project-context.md
- openspec/changes/<name>/context/existing-patterns.md
- openspec/changes/<name>/context/tech-constraints.md

## 调研任务

### 0. 自主方案探索（v3.0 新增）
在执行具体调研任务之前，先基于 RAW_REQUIREMENT 自主思考：
- 这个需求的核心技术挑战是什么？
- 业界是否有成熟的解决方案或最佳实践？
- 是否存在可以直接复用的开源组件或库？

如果判断需要联网搜索（depth >= standard），主动搜索：
- "{核心技术点} best practices {当前年份}"
- "{框架名} {功能类型} production experience"
- 在搜索结果中提取可行方案，形成推荐列表

产出：
- 推荐方案（含理由和数据支撑）
- 备选方案（含优劣对比）
- 不推荐方案（含排除理由）

### 1. 相关代码影响分析
- 搜索与需求相关的现有代码文件
- 估算需要新增/修改的文件数量和代码行数
- 识别可复用的现有模块或组件

### 2. 依赖兼容性检查
- 需求是否需要引入新依赖？
- 新依赖与现有依赖是否存在版本冲突？
- 是否存在已有依赖可以直接使用？

### 3. 技术可行性评估
- 当前技术栈是否支持需求中的所有功能？
- 是否存在技术障碍或已知限制？
- 推荐的技术方案（如有多个，列出优劣对比）

### 4. 风险识别
- 对现有功能的潜在破坏性影响
- 性能影响预估
- 安全隐患

### 5. Web 搜索调研（v2.4.0 新增，depth ≥ standard 时执行）
当 `config.phases.requirements.research.web_search !== false` 时：
- 使用 WebSearch 搜索 "{技术栈} + {功能需求关键词} + best practices 2026"
- 搜索 "{依赖名} + breaking changes + migration guide"（当需要引入新依赖时）
- 搜索 "{框架名} + {模式名} + alternatives comparison"
- 最多执行 config.phases.requirements.research.max_search_rounds 轮（默认 3 轮）
- 首轮概览 → 第二轮深化（基于首轮发现） → 第三轮验证

### 6. 同类实现对比（v2.4.0 新增，depth = deep 时执行）
- 搜索 GitHub/npm/Maven 上的同类功能实现
- 对比至少 2 种实现方案的架构差异
- 提取可复用的最佳实践

### 7. 依赖深度分析（v2.4.0 新增，depth = deep 时执行）
- 对每个候选新依赖：
  - 检查 npm/Maven 下载量和最近更新时间
  - 检查是否有已知安全漏洞（CVE）
  - 评估维护活跃度（最近 commit、open issues 数量）
  - 检查许可证兼容性

### 调研深度配置（v2.4.0 新增）

| 深度 | 触发条件 | 包含的调研任务 |
|------|---------|-------------|
| basic | `research.depth = "basic"` 或 complexity = "small" | 任务 1-4（现有） |
| standard | 默认，complexity = "medium" | 任务 1-5（含 Web 搜索） |
| deep | complexity = "large" 或 `research.depth = "deep"` | 任务 1-7（全部） |

### 1.3.3 联网调研（Web Research）

> 当 `config.phases.requirements.research.web_search.enabled = true`（standard/deep depth 默认 true，basic 默认跳过）时执行。

#### 1. 技术方案搜索

使用 WebSearch 搜索以下内容：
- 特定技术组合的最佳实践（如 "Vue 3 + WebSocket real-time collaboration best practices 2026"）
- GitHub 上的同类开源实现
- 已知的坑点和性能注意事项

#### 2. 依赖评估

对调研中识别的每个新依赖：
- WebSearch 搜索安全公告、维护状态、bundle size
- 对比替代方案（如 "Y.js vs Automerge vs ShareDB comparison 2026"）

#### 3. 产出

将联网调研结果写入 `research-findings.md` 的 Web Research Findings 章节，格式如下：

```markdown
## Web Research Findings

### Best Practices
- [source_url] pattern_name: description (relevance: high/medium/low)

### Similar Implementations
- [repo_url] approach: pros/cons

### Dependency Evaluation
- package_name: security_status, maintenance_status, bundle_size, alternatives

### Recommended Approach
Based on research, the recommended technical approach is: ...
```

#### 4. 搜索策略

| 深度 | 搜索行为 |
|------|---------|
| basic | 跳过联网搜索 |
| standard | 2-3 次定向搜索（最佳实践 + 关键依赖评估） |
| deep | 5+ 次搜索（含替代方案对比、安全审计、社区反馈） |

搜索次数上限由 `config.phases.requirements.research.web_search.max_queries`（默认 5）控制。

搜索关键词构造规则：
- 首轮：`"{技术栈} + {功能关键词} + best practices {当前年份}"`
- 次轮：基于首轮发现深化，如 `"{依赖名} + security advisory + CVE"`
- 深度轮：`"{方案A} vs {方案B} + benchmark + production experience"`

## 返回格式（JSON）
{
  "impact_analysis": {
    "new_files": ["predicted file paths"],
    "modified_files": ["existing file paths"],
    "estimated_loc": 数字,
    "reusable_modules": ["module names"]
  },
  "dependency_check": {
    "new_deps": [{"name": "...", "version": "...", "reason": "..."}],
    "conflicts": [],
    "existing_usable": ["dep names"]
  },
  "feasibility": {
    "score": "high|medium|low",
    "blockers": [],
    "recommendations": [{"approach": "...", "pros": [...], "cons": [...]}]
  },
  "risks": [{"category": "...", "description": "...", "severity": "high|medium|low", "mitigation": "..."}],
  "web_research": {
    "queries_executed": ["搜索查询 1", "搜索查询 2"],
    "best_practices": [{"source": "url", "pattern": "...", "relevance": "high|medium"}],
    "similar_implementations": [{"repo": "...", "approach": "...", "pros": ["..."], "cons": ["..."]}]
  }
}
```

### 返回值校验

主线程检查 Research Agent 返回：
- 非空 JSON
- 包含 `impact_analysis`、`feasibility` 字段
- 如果返回为空或格式异常，重新 dispatch 并在 prompt 中明确要求 JSON 输出
- 如果两次 dispatch 均失败，标记 `research_status: "skipped"`，继续流程（研究是增强而非阻断）

将调研结果写入 `openspec/changes/<name>/context/research-findings.md`。

## 1.4 复杂度评估与分路（Complexity Routing）

> 当 `config.phases.requirements.complexity_routing.enabled`（默认 true）时执行。

**目的**：根据调研结果自动评估复杂度，决定讨论深度。

### 评估规则

基于 Research Agent 返回的 `impact_analysis` 计算复杂度：

```
total_files = len(new_files) + len(modified_files)

IF total_files <= config.phases.requirements.complexity_routing.thresholds.small (默认 2):
    complexity = "small"
ELIF total_files <= config.phases.requirements.complexity_routing.thresholds.medium (默认 5):
    complexity = "medium"
ELSE:
    complexity = "large"
```

### 额外调整因子

即使文件数少，以下情况自动升级为 `"large"`：
- `feasibility.score == "low"`
- `risks` 中存在 `severity: "high"` 的风险
- 需要引入 3+ 个新依赖

### 分路策略

| 复杂度 | 讨论深度 | 苏格拉底模式 | 预计 QA 轮数 |
|--------|----------|-------------|-------------|
| small | 快速确认 — 展示调研结论，用户确认即可 | 禁用（即使 config 启用） | 1 轮 |
| medium | 标准讨论 — 完整决策循环 | 遵循 config 设置 | 2-3 轮 |
| large | 深度讨论 — 强制苏格拉底模式 | 强制启用（覆盖 config） | 3+ 轮 |

### 降级处理

如果 Research Agent 被跳过（research_status: "skipped"），默认 complexity = `"medium"`。

与旧版不同，business-analyst Agent 的 prompt 中**必须注入**以下上下文：

```
你是 autopilot Phase 1 的需求分析 Agent。

## 原始需求

读取以下文件了解项目背景：
- openspec/changes/<name>/context/project-context.md
- openspec/changes/<name>/context/existing-patterns.md
- openspec/changes/<name>/context/tech-constraints.md

## 技术调研结论

- openspec/changes/<name>/context/research-findings.md

## 1.5.5 结构化决策协议（Decision Protocol，v2.4.0 新增）

所有决策点**必须**以结构化卡片格式呈现给用户（complexity = medium 或 large 时强制）。

### 决策卡片格式

```markdown
## 决策 #{n}: {决策点标题}

**背景**: {为什么需要这个决策，引用 research-findings.md 中的事实数据}

| 选项 | 描述 | 优点 | 缺点 | 影响范围 |
|------|------|------|------|----------|
| A | {方案 A} | {pros} | {cons} | {影响的文件/模块} |
| B | {方案 B} | {pros} | {cons} | {影响的文件/模块} |
| C (Recommended) | {方案 C} | {pros} | {cons} | {影响的文件/模块} |

**推荐理由**: {基于调研数据的推荐依据}
**如果不决策**: {不做决策的默认行为和风险}
```

### 应用场景

| 场景 | 决策协议行为 |
|------|------------|
| Phase 1 需求决策（medium/large） | 完整决策卡片（含所有字段） |
| Phase 1 需求决策（small） | 简化模式 — 合并为一次确认 |
| Phase 5 合并冲突 | 简化决策卡片（仅选项 + 影响范围） |
| Phase 7 归档确认 | 单选确认（现有行为不变） |

### 决策记录持久化

每个决策点记录到 checkpoint 的 `decisions` 数组，格式为 DecisionPoint（见 protocol.md）：
- 包含 `options`（含 pros/cons/recommended）
- 包含 `rationale`（用户选择理由）
- 包含 `affected_components`（影响范围）

> **向后兼容**: complexity = "small" 或旧配置无此功能时，退回标准 AskUserQuestion 格式。

## 1.6 多轮决策循环（LOOP）

**循环条件**: 存在任何未澄清的决策点

### 主动讨论协议（v3.0 新增）

**核心原则**：但凡存在不确定性，主动列出可选方案，由用户决策。

对每个识别到的不确定点，编排器必须：

1. **结构化呈现**决策卡片：
   - 列出 2-4 个可选方案
   - 每个方案标注优点、缺点、影响范围
   - 标记推荐方案（基于调研数据）
   - 说明"如果不决策"的默认行为和风险

2. **分类处理**：
   - **技术决策**（框架选择、架构模式）→ 完整决策卡片 + 调研数据支撑
   - **范围决策**（功能边界、MVP 范围）→ 优先级矩阵 + 工期影响
   - **风险决策**（安全、性能、兼容性）→ 风险评估表 + 缓解方案

3. **等待用户选择**：通过 AskUserQuestion 呈现，绝不自行决策

### 基础流程（所有复杂度）

每轮循环:
1. 梳理当前所有未决策点
2. 将每个决策点构造为决策卡片格式（见 1.5.5 节），通过 AskUserQuestion 展示（2-4 个选项，推荐方案标 Recommended）
3. 收集用户决策结果，记录 rationale
4. 检查是否产生新的决策点
5. 重复直到**所有点全部澄清**

### Small 复杂度快速路径

当 complexity == "small" 时：
1. 将所有决策点合并为一次 AskUserQuestion（多选模式或逐一快速确认）
2. 展示调研结论 + 推荐方案，让用户一次性确认
3. 最多 1 轮循环，用户确认即退出

### Large 复杂度强化路径

当 complexity == "large" 时，强制附加以下检查：
1. 每轮循环后检查是否有需求蔓延（scope creep）迹象
2. 如果功能点持续增加，主动提出 MVP 范围收敛建议
3. 必须至少完成 3 轮 QA 才能退出循环

## 1.7 生成结构化提示词

整理所有决策结果，包含：

| 章节 | 内容 |
|------|------|
| 背景与目标 | 需求来源 + 业务目标 |
| 功能清单 | 确认的功能点列表（含优先级） |
| 决策结论 | 每个决策点的最终选择 |
| 技术约束 | 从 tech-constraints.md 提取 |
| 技术方案 | 从 research-findings.md 提取推荐方案 |
| 验收标准 | 可测试的验收条件 |
| 影响范围 | 预估影响文件 + 代码行数 |

## 1.8 最终确认

展示完整提示词，AskUserQuestion:
"以上需求理解是否准确？如有遗漏请补充。"
选项: "确认，开始实施 (Recommended)" / "需要补充修改"
- 选"补充" → 回到 1.6 循环

## 1.9 写入 Phase 1 Checkpoint

需求确认后，调用 Skill(`spec-autopilot:autopilot-checkpoint`) 写入 `phase-1-requirements.json`：

```json

## 苏格拉底模式协议

> 当 `config.phases.requirements.mode` 设为 `"socratic"` 时激活（默认 `"structured"`）。
> 当 complexity == "large" 时**强制激活**（覆盖 config 设置）。

苏格拉底模式通过挑战性提问深化需求理解，适用于需求模糊或创新性项目。

### 6 步提问流程

| 步骤 | 目的 | 示例提问 |
|------|------|---------|
| 1. 挑战假设 | 质疑用户隐含的前提条件 | "你提到需要实时同步，是否考虑过最终一致性就能满足需求？" |
| 2. 探索替代方案 | 寻找用户未考虑的解决路径 | "除了新增 API 端点，是否可以通过扩展现有端点的参数来实现？" |
| 3. 识别隐含需求 | 挖掘用户未明确表达的需求 | "你提到了创建功能，但是否还需要批量导入？删除时是否需要软删除？" |
| 4. 强制排优 | 当功能过多时迫使用户取舍 | "这 5 个功能中，如果只能在第一个版本实现 3 个，你会选择哪些？" |
| 5. 魔鬼代言人 | 从反对角度审视需求 | "如果竞品已经有这个功能，我们的差异化价值在哪里？" |
| 6. 最小可行范围 | 收敛到最小可交付范围 | "基于以上讨论，最小可行版本是否可以只包含 X 和 Y？" |

### 集成方式

在 Phase 1.6 多轮决策循环中，每轮除了处理用户疑问外，额外执行苏格拉底提问：

```
IF config.phases.requirements.mode == "socratic" OR complexity == "large":
    FOR EACH undecided_point:
        1. 标准处理：AskUserQuestion 展示选项
        2. 苏格拉底追问：从 6 步中选择最相关的 1-2 步提问
        3. 引用 research-findings.md 中的事实数据支撑提问
        4. 根据用户回答，可能产生新的决策点（回到循环开头）
```

### 退出条件

- 所有决策点已澄清 **且** 苏格拉底提问未产生新决策点
- 用户明确表示"需求已经足够清晰，不需要更多提问"

## 崩溃恢复行为

Phase 1 崩溃恢复时的特殊处理：

| 已完成步骤 | 恢复行为 |
|-----------|---------|
| 1.2 Steering 文件已存在 | 跳过扫描，直接读取 |
| 1.3 research-findings.md 已存在 | 跳过调研，直接读取 |
| 1.4 复杂度已评估 | 从 checkpoint 恢复 complexity 值 |
| 1.5 需求分析已完成 | 跳过分析，直接进入决策循环 |
| 1.9 Checkpoint 已写入 | Phase 1 视为完成，跳过整个阶段 |

### 决策格式 Hook 强制验证

Phase 1 子 Agent 返回的 JSON 信封中的 `decisions` 数组现在由 `validate-decision-format.sh` Hook 确定性验证：

| 复杂度 | 验证规则 |
|--------|----------|
| medium / large | 完整 DecisionPoint 格式：每个决策必须包含 `options`（≥2 个选项，每个含 `label`/`description`/`pros`/`cons`），至少一个 `recommended: true`，以及 `choice` 和 `rationale` 字段 |
| small | 简化格式：旧版 `{point, choice}` 格式可接受，仅检查基本字段存在性 |

验证失败时 Hook 输出 `decision: "block"`，列出具体缺失字段，阻止流水线继续。

这确保了所有决策点都有充分的选项分析，用户可以做出知情决策。
