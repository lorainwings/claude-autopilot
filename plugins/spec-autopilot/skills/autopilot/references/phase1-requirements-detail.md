## Auto-Scan 扫描范围

使用 Glob + Grep + Read 在主线程中扫描以下信息：

| 扫描项 | 方法 | 产出 |
|--------|------|------|
| 技术栈 | Glob: `**/build.gradle*`, `**/pom.xml`, `**/package.json`, `**/go.mod`, `**/Cargo.toml`, `**/requirements.txt` | 语言 + 框架 + 版本 |
| 目录结构 | Glob: `**/src/**`, 结合 `.gitignore` 排除 | 模块布局树（2 层深度） |
| 现有模式 | Grep: 路由定义、API 端点、数据库模型、组件注册 | 已有代码模式清单 |
| 依赖清单 | Read: `package.json`, `build.gradle`, `requirements.txt` 的依赖节 | 关键依赖 + 版本 |
| 项目约束 | Read: `CLAUDE.md`, `.claude/rules/*.md`（前 50 行） | 编码约束清单 |
| 测试基础设施 | Glob: `**/test*/**`, `**/spec*/**`, `**/*.test.*`, `**/*.spec.*` | 现有测试框架 + 覆盖范围 |

## Steering Documents 模板

扫描结果写入 `openspec/changes/<name>/context/` 目录下的 3 个 Steering 文件：

**`project-context.md`**（项目上下文）：
```markdown
# Project Context — Auto-generated by Autopilot Phase 1

## Tech Stack
- Backend: {language} {version} + {framework} {version}
- Frontend: {framework} {version} + {bundler}
- Database: {type}

## Directory Layout
{2-level module tree}

## Key Dependencies
{top 10 dependencies with versions}

## Coding Constraints (from CLAUDE.md / rules)
{extracted constraints list}

## Existing Test Infrastructure
{test framework, test count, coverage baseline}
```

**`existing-patterns.md`**（现有代码模式）：
```markdown
# Existing Patterns — Auto-generated by Autopilot Phase 1

## API Patterns
{route definitions, naming conventions, auth middleware}

## Data Models
{entity list with key fields}

## Component Patterns (Frontend)
{component naming, state management, routing}

{error handling approach, common patterns}
```

**`tech-constraints.md`**（技术约束）：
```markdown
# Technical Constraints — Auto-generated by Autopilot Phase 1

{extracted from CLAUDE.md and .claude/rules/}

{version pinning, peer dependency requirements}

{deployment target, environment requirements}
```

## 历史知识注入（v2.4.0 新增）

Auto-Scan 完成后，检查 `openspec/.autopilot-knowledge.json` 是否存在：

1. **存在** → 读取 entries 数组，按 tags 与 RAW_REQUIREMENT 关键词匹配
2. 取 top-5 相关度最高的条目
3. 写入 `project-context.md` 的「Historical Knowledge」章节：
   ```markdown
   ## Historical Knowledge (from previous autopilot sessions)

   - [pitfall] {summary}: {detail} (from change: {change_name})
   - [decision] {summary}: {detail} (from change: {change_name})
   - [pattern] {summary}: {detail} (reused {reuse_count} times)
   ```
4. 匹配到的 entries 的 `reuse_count += 1`（延迟写回到 knowledge.json）
5. **不存在** → 跳过（首次运行无历史数据）

> 详见：`references/knowledge-accumulation.md`

## Research Agent Prompt 模板

```
你是 autopilot Phase 1 的技术调研 Agent。

## 需求概述
{RAW_REQUIREMENT}

## 项目上下文
读取以下文件获取项目背景：
- openspec/changes/<name>/context/project-context.md
- openspec/changes/<name>/context/existing-patterns.md
- openspec/changes/<name>/context/tech-constraints.md

## 调研任务

### 0. 自主方案探索（v3.0 新增）
在执行具体调研任务之前，先基于 RAW_REQUIREMENT 自主思考：
- 这个需求的核心技术挑战是什么？
- 业界是否有成熟的解决方案或最佳实践？
- 是否存在可以直接复用的开源组件或库？

如果判断需要联网搜索（depth >= standard），主动搜索：
- "{核心技术点} best practices {当前年份}"
- "{框架名} {功能类型} production experience"
- 在搜索结果中提取可行方案，形成推荐列表

产出：
- 推荐方案（含理由和数据支撑）
- 备选方案（含优劣对比）
- 不推荐方案（含排除理由）

### 1. 相关代码影响分析
- 搜索与需求相关的现有代码文件
- 估算需要新增/修改的文件数量和代码行数
- 识别可复用的现有模块或组件

### 2. 依赖兼容性检查
- 需求是否需要引入新依赖？
- 新依赖与现有依赖是否存在版本冲突？
- 是否存在已有依赖可以直接使用？

### 3. 技术可行性评估
- 当前技术栈是否支持需求中的所有功能？
- 是否存在技术障碍或已知限制？
- 推荐的技术方案（如有多个，列出优劣对比）

### 4. 风险识别
- 对现有功能的潜在破坏性影响
- 性能影响预估
- 安全隐患

### 5. Web 搜索调研（v2.4.0 新增, v3.3.7 策略重构）
默认执行搜索（`search_policy.default: search`），仅当规则引擎判定 skip 时跳过。详见搜索决策规则引擎。
当未被跳过时：
- 使用 WebSearch 搜索 "{技术栈} + {功能需求关键词} + best practices 2026"
- 搜索 "{依赖名} + breaking changes + migration guide"（当需要引入新依赖时）
- 搜索 "{框架名} + {模式名} + alternatives comparison"
- 最多执行 config.phases.requirements.research.max_search_rounds 轮（默认 3 轮）
- 首轮概览 → 第二轮深化（基于首轮发现） → 第三轮验证

### 6. 同类实现对比（v2.4.0 新增，depth = deep 时执行）
- 搜索 GitHub/npm/Maven 上的同类功能实现
- 对比至少 2 种实现方案的架构差异
- 提取可复用的最佳实践

### 7. 依赖深度分析（v2.4.0 新增，depth = deep 时执行）
- 对每个候选新依赖：
  - 检查 npm/Maven 下载量和最近更新时间
  - 检查是否有已知安全漏洞（CVE）
  - 评估维护活跃度（最近 commit、open issues 数量）
  - 检查许可证兼容性
```

### 调研深度配置（v2.4.0 新增）

| 深度 | 触发条件 | 包含的调研任务 |
|------|---------|-------------|
| basic | `research.depth = "basic"` 或 complexity = "small" | 任务 1-4（现有） |
| standard | 默认，complexity = "medium" | 任务 1-5（含 Web 搜索） |
| deep | complexity = "large" 或 `research.depth = "deep"` | 任务 1-7（全部） |

## 搜索决策规则引擎（v3.3.7）

**核心原则**: 默认执行搜索，仅当规则引擎判定可跳过时才跳过。不依赖 AI 自评置信度。

### 规则定义

```
search_policy:
  default: search    # 默认搜索，跳过是例外

  # 跳过条件（必须全部满足才跳过）
  skip_when_ALL_true:
    - task_type ∈ {bugfix, refactor, style_adjustment}     # 纯内部变更
    - no_new_concepts: true                                  # 不引入新概念/模式/交互
    - rules_coverage: project rules/ 或 specs/ 已有规范覆盖  # 本地知识充足
    - has_precedent: codebase 中已有同类实现                  # 有先例可参照

  # 以下任一为 true 时强制搜索（覆盖 skip 判定）
  force_search_when_ANY_true:
    - new_feature: true          # 新功能/新页面/新交互
    - new_dependency: true       # 引入项目未用过的依赖
    - ux_improvement: true       # UX 改进（需竞品参考）
    - architecture_decision: true # 架构决策（需方案对比）
    - security_related: true     # 安全相关
    - version_upgrade: true      # 框架/依赖版本升级
    - product_design: true       # 产品设计（需竞品 UX 参考）
```

### 规则执行方式（非 AI 自评）

每条规则通过**可编程检测**执行：

| 规则 | 检测方式 | AI 参与度 |
|------|---------|----------|
| task_type | 从需求文本匹配关键词: "修复/fix/重构/refactor/样式/style" | 低（关键词提取） |
| new_dependency | `grep <lib> package.json build.gradle` — 不存在即为新依赖 | 零 |
| rules_coverage | `ls rules/ specs/` 检查是否有对应领域文件 | 零 |
| has_precedent | `grep -r <pattern> src/` 检查已有同类实现 | 低（关键词提取） |
| ux_improvement | 需求含 "UX/交互/体验/界面优化" 关键词 | 低 |
| product_design | 需求含 "竞品/产品/用户体验/参考" 关键词 | 低 |
| security_related | 需求含 "auth/加密/CORS/XSS/JWT/安全" 关键词 | 零 |
| version_upgrade | 需求含 "升级/迁移/v\d+/migration" 关键词 | 零 |

### 搜索分类

| 搜索类型 | 目的 | 典型查询 | 价值 |
|---------|------|---------|------|
| **产品搜索** | 竞品 UX、交互模式、行业趋势 | "Cursor AI slash command UX", "Linear keyboard shortcuts" | 不可替代（AI 训练数据不覆盖） |
| **技术搜索** | API 文档、框架配置、breaking changes | "Spring Boot 3.5 WebSocket config" | 补充性（AI 内置知识可能过时） |

产品搜索价值 > 技术搜索，因为**竞品实现是 AI 无法从自身知识推导出来的**。

### 搜索深度（与 complexity 联动）

| 深度 | 触发条件 | 搜索行为 |
|------|---------|---------|
| basic | `complexity = "small"` 且规则判定可跳过 | 跳过联网搜索 |
| standard | 默认 | 2-3 次定向搜索（竞品参考 + 最佳实践 + 关键依赖评估） |
| deep | `complexity = "large"` 或 `research.depth = "deep"` | 5+ 次搜索（含替代方案对比、安全审计、竞品深度分析） |

搜索次数上限由 `config.phases.requirements.research.web_search.max_queries`（默认 5）控制。

### 搜索关键词构造规则

- **产品搜索**：`"{竞品名} + {功能类型} + UX design {当前年份}"`
- **技术搜索首轮**：`"{技术栈} + {功能关键词} + best practices {当前年份}"`
- **技术搜索次轮**：基于首轮发现深化，如 `"{依赖名} + security advisory + CVE"`
- **深度轮**：`"{方案A} vs {方案B} + benchmark + production experience"`

### 搜索结果信任机制

搜索结果**不可盲信**，必须交叉验证：

1. **标注来源**：每条结论标注 URL + 日期
2. **交叉验证**：搜索结果 vs 项目 `rules/`、`specs/`、已有代码对比
3. **冲突解决**：
   - 搜索结果 vs 项目规范冲突 → **以项目规范为准**
   - 搜索结果 vs AI 内置知识冲突 → 标记为"需用户确认"
4. **降级策略**：搜索失败/超时 → 回退到 AI 内置知识，不阻塞流程

## Research Agent 返回格式（完整 JSON 信封）

```json
{
  "status": "ok",
  "summary": "简明调研结论（3-5句）",
  "decision_points": [
    {"topic": "决策点标题", "options": ["方案A", "方案B"], "recommendation": "推荐方案", "rationale": "推荐理由"}
  ],
  "tech_constraints": ["硬性约束1", "硬性约束2"],
  "complexity": "small|medium|large",
  "key_files": ["关键文件路径"],
  "output_file": "context/research-findings.md",
  "impact_analysis": {
    "new_files": ["predicted file paths"],
    "modified_files": ["existing file paths"],
    "estimated_loc": 数字,
    "reusable_modules": ["module names"]
  },
  "feasibility": {
    "score": "high|medium|low",
    "blockers": [],
    "recommendations": [{"approach": "...", "pros": [...], "cons": [...]}]
  },
  "risks": [{"category": "...", "severity": "high|medium|low", "mitigation": "..."}]
}
```

> **字段说明**：
> - `decision_points`：提取的技术/产品决策点，主线程直接用于构造决策卡片
> - `tech_constraints`：从项目规则和代码中发现的硬性约束
> - `complexity`：基于 impact_analysis 自评的复杂度
> - `output_file`：调研全文写入的文件路径（供主线程验证文件存在）
