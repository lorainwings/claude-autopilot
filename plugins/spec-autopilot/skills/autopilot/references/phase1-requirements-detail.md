## 1.2 Auto-Scan 详细流程

### 持久化上下文检查（优先于全量扫描）

在执行扫描前，先检查项目级持久化上下文：

```
persistent_dir = "openspec/.autopilot-context/"

IF persistent_dir exists AND project-context.md exists:
    mtime = file modification time of project-context.md
    IF (now - mtime) < 7 days:
        # 新鲜 → 跳过全量扫描，直接读取
        READ persistent_dir/project-context.md → 注入上下文
        READ persistent_dir/existing-patterns.md → 注入模式
        READ persistent_dir/tech-constraints.md → 注入约束
        # 仅对本次需求相关代码做增量扫描（Grep 需求关键词相关文件）
        SKIP 全量 Steering Documents 生成
    ELSE:
        # 过期 → 增量更新
        READ existing files → 作为基线
        RUN 全量扫描 → 与基线 diff
        UPDATE persistent files with delta
ELSE:
    # 首次 → 全量扫描（当前行为）
    RUN 全量扫描
    WRITE results to persistent_dir (new)
    COPY to per-change context/
```

> **幂等性**：持久化文件更新采用读取 → 修改 → 完整写回策略，非追加写入。

### 扫描范围

使用 Glob + Grep + Read 在主线程中扫描以下信息：

| 扫描项 | 方法 | 产出 |
|--------|------|------|
| 技术栈 | Glob: `**/build.gradle*`, `**/pom.xml`, `**/package.json`, `**/go.mod`, `**/Cargo.toml`, `**/requirements.txt` | 语言 + 框架 + 版本 |
| 目录结构 | Glob: `**/src/**`, 结合 `.gitignore` 排除 | 模块布局树（2 层深度） |
| 现有模式 | Grep: 路由定义、API 端点、数据库模型、组件注册 | 已有代码模式清单 |
| 依赖清单 | Read: `package.json`, `build.gradle`, `requirements.txt` 的依赖节 | 关键依赖 + 版本 |
| 项目约束 | Read: `CLAUDE.md`, `.claude/rules/*.md`（前 50 行） | 编码约束清单 |
| 测试基础设施 | Glob: `**/test*/**`, `**/spec*/**`, `**/*.test.*`, `**/*.spec.*` | 现有测试框架 + 覆盖范围 |

### 产出 Steering Documents

扫描结果写入 `openspec/changes/<name>/context/` 目录下的 3 个 Steering 文件：

**`project-context.md`**（项目上下文）：
```markdown
# Project Context — Auto-generated by Autopilot Phase 1

## Tech Stack
- Backend: {language} {version} + {framework} {version}
- Frontend: {framework} {version} + {bundler}
- Database: {type}

## Directory Layout
{2-level module tree}

## Key Dependencies
{top 10 dependencies with versions}

## Coding Constraints (from CLAUDE.md / rules)
{extracted constraints list}

## Existing Test Infrastructure
{test framework, test count, coverage baseline}
```

### 历史知识注入

Auto-Scan 完成后，检查 `openspec/.autopilot-knowledge.json` 是否存在：

1. **存在** → 读取 entries 数组，按 tags 与 RAW_REQUIREMENT 关键词匹配
2. 取 top-5 相关度最高的条目
3. 写入 `project-context.md` 的「Historical Knowledge」章节：
   ```markdown
   ## Historical Knowledge (from previous autopilot sessions)

   - [pitfall] {summary}: {detail} (from change: {change_name})
   - [decision] {summary}: {detail} (from change: {change_name})
   - [pattern] {summary}: {detail} (reused {reuse_count} times)
   ```
4. 匹配到的 entries 的 `reuse_count += 1`（延迟写回到 knowledge.json）
5. **不存在** → 跳过（首次运行无历史数据）

> 详见：`references/knowledge-accumulation.md`

**`existing-patterns.md`**（现有代码模式）：
```markdown
# Existing Patterns — Auto-generated by Autopilot Phase 1

## API Patterns
{route definitions, naming conventions, auth middleware}

## Data Models
{entity list with key fields}

## Component Patterns (Frontend)
{component naming, state management, routing}

## Error Handling
{error handling approach, common patterns}
```

**`tech-constraints.md`**（技术约束）：
```markdown
# Technical Constraints — Auto-generated by Autopilot Phase 1

## Hard Constraints (from project rules)
{extracted from CLAUDE.md and .claude/rules/}

## Dependency Constraints
{version pinning, peer dependency requirements}

## Infrastructure Constraints
{deployment target, environment requirements}
```

> **幂等性**：如果 Steering 文件已存在（崩溃恢复场景），跳过扫描，直接读取已有文件。

### Auto-Scan 返回信封（上下文保护）

Auto-Scan Agent 完成 Steering Documents 写入后，返回 JSON 信封（禁止返回文件全文）：

```json
{
  "status": "ok",
  "summary": "项目概况摘要（2-3句）",
  "tech_constraints": ["从 CLAUDE.md / rules 提取的硬性约束"],
  "key_files": ["核心文件路径"],
  "output_files": ["context/project-context.md", "context/existing-patterns.md", "context/tech-constraints.md"]
}
```

> 主线程仅验证 `output_files` 指向的文件存在，不 Read 文件内容。

## 1.3 Research Agent 完整 Prompt 模板

```
你是 autopilot Phase 1 的技术调研 Agent。

## 需求概述
{RAW_REQUIREMENT}

## 项目上下文
读取以下文件获取项目背景：
- openspec/changes/<name>/context/project-context.md
- openspec/changes/<name>/context/existing-patterns.md
- openspec/changes/<name>/context/tech-constraints.md

## 调研任务

### 0. 自主方案探索
在执行具体调研任务之前，先基于 RAW_REQUIREMENT 自主思考：
- 这个需求的核心技术挑战是什么？
- 业界是否有成熟的解决方案或最佳实践？
- 是否存在可以直接复用的开源组件或库？

如果判断需要联网搜索（depth >= standard），主动搜索：
- "{核心技术点} best practices {当前年份}"
- "{框架名} {功能类型} production experience"
- 在搜索结果中提取可行方案，形成推荐列表

产出：
- 推荐方案（含理由和数据支撑）
- 备选方案（含优劣对比）
- 不推荐方案（含排除理由）

### 1. 相关代码影响分析
- 搜索与需求相关的现有代码文件
- 估算需要新增/修改的文件数量和代码行数
- 识别可复用的现有模块或组件

### 2. 依赖兼容性检查
- 需求是否需要引入新依赖？
- 新依赖与现有依赖是否存在版本冲突？
- 是否存在已有依赖可以直接使用？

### 3. 技术可行性评估
- 当前技术栈是否支持需求中的所有功能？
- 是否存在技术障碍或已知限制？
- 推荐的技术方案（如有多个，列出优劣对比）

### 4. 风险识别
- 对现有功能的潜在破坏性影响
- 性能影响预估
- 安全隐患

### 5. Web 搜索调研（depth = deep 时执行）
> **触发条件**：当且仅当 `depth = deep`（complexity == "large" / maturity == "ambiguous" / `research.depth == "deep"` 任一命中即升级）。`depth = standard` 时 `websearch_quota = 0`，本任务不执行；越权调用由 L2 Hook 阻断。
>
> 已合并自 v5.x 第 3 路独立 web_search Agent，作为 ResearchAgent 的条件子任务执行。
- 使用 WebSearch 搜索 "{技术栈} + {功能需求关键词} + best practices 2026"
- 搜索 "{依赖名} + breaking changes + migration guide"（当需要引入新依赖时）
- 搜索 "{框架名} + {模式名} + alternatives comparison"
- 最多执行 config.phases.requirements.research.max_search_rounds 轮（默认 3 轮）
- 首轮概览 → 第二轮深化（基于首轮发现） → 第三轮验证
- 调研结果直接合并到 `research-findings.md` 的「Web Research Findings」章节，不再单独输出 `web-research-findings.md`

### 6. 同类实现对比（depth = deep 时执行）
- 搜索 GitHub/npm/Maven 上的同类功能实现
- 对比至少 2 种实现方案的架构差异
- 提取可复用的最佳实践

### 7. 依赖深度分析（depth = deep 时执行）
- 对每个候选新依赖：
  - 检查 npm/Maven 下载量和最近更新时间
  - 检查是否有已知安全漏洞（CVE）
  - 评估维护活跃度（最近 commit、open issues 数量）
  - 检查许可证兼容性
```

### 调研深度配置

| 深度 | 触发条件 | 包含的调研任务 |
|------|---------|-------------|
| basic | `research.depth = "basic"` 或 complexity = "small" | 任务 1-4（现有） |
| standard | 默认，complexity = "medium" | 任务 1-5（含 Web 搜索） |
| deep | complexity = "large" 或 `research.depth = "deep"` | 任务 1-7（全部） |

## 1.3.3 搜索决策规则引擎完整规则

**核心原则**: 默认执行搜索，仅当规则引擎判定可跳过时才跳过。不依赖 AI 自评置信度。

### 规则定义

```
search_policy:
  default: search    # 默认搜索，跳过是例外

  # 跳过条件（必须全部满足才跳过）
  skip_when_ALL_true:
    - task_type ∈ {bugfix, refactor, style_adjustment}     # 纯内部变更
    - no_new_concepts: true                                  # 不引入新概念/模式/交互
    - rules_coverage: project rules/ 或 specs/ 已有规范覆盖  # 本地知识充足
    - has_precedent: codebase 中已有同类实现                  # 有先例可参照

  # 以下任一为 true 时强制搜索（覆盖 skip 判定）
  force_search_when_ANY_true:
    - new_feature: true          # 新功能/新页面/新交互
    - new_dependency: true       # 引入项目未用过的依赖
    - ux_improvement: true       # UX 改进（需竞品参考）
    - architecture_decision: true # 架构决策（需方案对比）
    - security_related: true     # 安全相关
    - version_upgrade: true      # 框架/依赖版本升级
    - product_design: true       # 产品设计（需竞品 UX 参考）
```

### 规则执行方式（非 AI 自评）

每条规则通过**可编程检测**执行：

| 规则 | 检测方式 | AI 参与度 |
|------|---------|----------|
| task_type | 从需求文本匹配关键词: "修复/fix/重构/refactor/样式/style" | 低（关键词提取） |
| new_dependency | `grep <lib> package.json build.gradle` — 不存在即为新依赖 | 零 |
| rules_coverage | `ls rules/ specs/` 检查是否有对应领域文件 | 零 |
| has_precedent | `grep -r <pattern> src/` 检查已有同类实现 | 低（关键词提取） |
| ux_improvement | 需求含 "UX/交互/体验/界面优化" 关键词 | 低 |
| product_design | 需求含 "竞品/产品/用户体验/参考" 关键词 | 低 |
| security_related | 需求含 "auth/加密/CORS/XSS/JWT/安全" 关键词 | 零 |
| version_upgrade | 需求含 "升级/迁移/v\d+/migration" 关键词 | 零 |

**唯一需要 AI 做的是从需求文本提取关键词**（技术名、库名），这一步 LLM 可靠——提取事实比评估自身知识容易得多。

### 搜索分类

| 搜索类型 | 目的 | 典型查询 | 价值 |
|---------|------|---------|------|
| **产品搜索** | 竞品 UX、交互模式、行业趋势 | "Cursor AI slash command UX", "Linear keyboard shortcuts" | 不可替代（AI 训练数据不覆盖） |
| **技术搜索** | API 文档、框架配置、breaking changes | "Spring Boot 3.5 WebSocket config" | 补充性（AI 内置知识可能过时） |

产品搜索价值 > 技术搜索，因为**竞品实现是 AI 无法从自身知识推导出来的**。

### 搜索深度（与 complexity 联动）

| 深度 | 触发条件 | 搜索行为 |
|------|---------|---------|
| basic | `complexity = "small"` 且规则判定可跳过 | 跳过联网搜索 |
| standard | 默认 | 2-3 次定向搜索（竞品参考 + 最佳实践 + 关键依赖评估） |
| deep | `complexity = "large"` 或 `research.depth = "deep"` | 5+ 次搜索（含替代方案对比、安全审计、竞品深度分析） |

搜索次数上限由 `config.phases.requirements.research.web_search.max_queries`（默认 5）控制。

### 搜索关键词构造规则

- **产品搜索**：`"{竞品名} + {功能类型} + UX design {当前年份}"`
- **技术搜索首轮**：`"{技术栈} + {功能关键词} + best practices {当前年份}"`
- **技术搜索次轮**：基于首轮发现深化，如 `"{依赖名} + security advisory + CVE"`
- **深度轮**：`"{方案A} vs {方案B} + benchmark + production experience"`

### 搜索结果信任机制

搜索结果**不可盲信**，必须交叉验证：

1. **标注来源**：每条结论标注 URL + 日期
2. **交叉验证**：搜索结果 vs 项目 `rules/`、`specs/`、已有代码对比
3. **冲突解决**：
   - 搜索结果 vs 项目规范冲突 → **以项目规范为准**
   - 搜索结果 vs AI 内置知识冲突 → 标记为"需用户确认"
4. **降级策略**：搜索失败/超时 → 回退到 AI 内置知识，不阻塞流程

### 联网调研产出（合并到 ResearchAgent 主产物）

> v6.x 起，联网调研结果**不再单独输出** `web-research-findings.md`，而是作为 ResearchAgent 主产物 `research-findings.md` 的一个子章节追加。这是 v5.x 第 3 路独立 web_search Agent 合并到 ResearchAgent 后的统一产物模型。

ResearchAgent 在 `depth = deep` 时，将联网调研结果合并写入 `openspec/changes/<name>/context/research-findings.md` 的 `## Web Research Findings` 章节，结构如下：

```markdown
## Web Research Findings (depth=deep only)

### Competitive Analysis（竞品分析）
- [source_url] competitor_name: UX pattern description (relevance: high/medium/low)

### Best Practices
- [source_url] pattern_name: description (relevance: high/medium/low)

### Similar Implementations
- [repo_url] approach: pros/cons

### Dependency Evaluation
- package_name: security_status, maintenance_status, bundle_size, alternatives

### Recommended Approach
Based on research, the recommended technical approach is: ...

### Source Trust Level
- Each finding marked: [verified|unverified|conflicts_with_project_rules]
```

联网调研摘要在 ResearchAgent 返回的 JSON 信封中以 `web_search_summary` 字段（仅当 depth=deep 时）回传，主线程不读取正文。

## Research Agent 产出写入与返回格式

### 产出写入（上下文保护）

**子 Agent 必须自行写入调研全文**，主线程不参与文件写入：

1. 将调研详情 Write 到 `openspec/changes/<name>/context/research-findings.md`
2. 返回 JSON 信封（仅摘要，禁止包含调研全文）

> **上下文保护原则**：调研全文仅存在于文件中，由需求分析 Agent 和后续 Phase 子 Agent 直接 Read。主线程仅消费信封中的结构化摘要。

### 返回格式（JSON 信封）

```json
{
  "status": "ok",
  "summary": "简明调研结论（3-5句）",
  "decision_points": [
    {"topic": "决策点标题", "options": ["方案A", "方案B"], "recommendation": "推荐方案", "rationale": "推荐理由"}
  ],
  "tech_constraints": ["硬性约束1", "硬性约束2"],
  "complexity": "small|medium|large",
  "key_files": ["关键文件路径"],
  "output_file": "context/research-findings.md",
  "impact_analysis": {
    "new_files": ["predicted file paths"],
    "modified_files": ["existing file paths"],
    "estimated_loc": 数字,
    "reusable_modules": ["module names"]
  },
  "feasibility": {
    "score": "high|medium|low",
    "blockers": [],
    "recommendations": [{"approach": "...", "pros": [...], "cons": [...]}]
  },
  "risks": [{"category": "...", "severity": "high|medium|low", "mitigation": "..."}],
  "web_search_summary": {
    "search_decision": "searched|skipped|not_applicable",
    "skip_reason": "仅当 skipped 时填写跳过原因；depth=standard 时为 not_applicable",
    "queries_executed": 0,
    "highlights": ["关键发现 URL + 一句话"]
  }
}
```

> **字段说明**：
> - `decision_points`：提取的技术/产品决策点，主线程直接用于构造决策卡片
> - `tech_constraints`：从项目规则和代码中发现的硬性约束
> - `complexity`：基于 impact_analysis 自评的复杂度
> - `output_file`：调研全文写入的文件路径（供主线程验证文件存在）
> - `web_search_summary.search_decision`：联网调研子任务执行状态（仅 depth=deep 时可能为 searched，否则为 not_applicable）

### 返回值校验

主线程检查 Research Agent 返回：
- 非空 JSON 信封
- 包含 `status`、`decision_points`、`feasibility`、`output_file` 字段
- 验证 `output_file` 指向的文件存在且非空：`Bash("test -s {output_file} && echo ok")`
- 如果返回为空或格式异常，重新 dispatch 并在 prompt 中明确要求 JSON 输出
- 如果两次 dispatch 均失败，标记 `research_status: "skipped"`，继续流程（研究是增强而非阻断）

## 1.4 复杂度评估完整规则

### 评估规则

基于 Research Agent 返回的 `impact_analysis` 计算复杂度：

```
total_files = len(new_files) + len(modified_files)

IF total_files <= config.phases.requirements.complexity_routing.thresholds.small (默认 2):
    complexity = "small"
ELIF total_files <= config.phases.requirements.complexity_routing.thresholds.medium (默认 5):
    complexity = "medium"
ELSE:
    complexity = "large"
```

### 额外调整因子

即使文件数少，以下情况自动升级为 `"large"`：
- `feasibility.score == "low"`
- `risks` 中存在 `severity: "high"` 的风险
- 需要引入 3+ 个新依赖

### 多维度复杂度修正

> **核心原则**: 不能仅依赖 total_files 判定复杂度。"少文件但高风险"不能误判为 small。

以下维度独立评估，任一命中即触发复杂度升级：

```
complexity_boost_flags = []

# 维度 1: 跨模块影响
IF modified_files 涉及 >= 2 个顶层目录（如 src/auth + src/payment + src/api）:
    complexity_boost_flags.append("cross_module")

# 维度 2: 高风险域
high_risk_domains = ["安全", "认证", "auth", "支付", "payment", "性能", "performance",
                     "迁移", "migration", "加密", "crypto", "数据库迁移", "schema"]
IF RAW_REQUIREMENT 或 impact_analysis 涉及 high_risk_domains 中的关键词:
    complexity_boost_flags.append("high_risk_domain")

# 维度 3: 新依赖引入
IF impact_analysis.new_dependencies >= 1 且依赖为项目未使用过的:
    complexity_boost_flags.append("new_dependency")

# 维度 4: 非功能约束
IF 需求涉及性能指标（延迟/吞吐/并发）、可用性要求、数据一致性、向后兼容:
    complexity_boost_flags.append("non_functional_constraint")

# 维度 5: 多决策点
IF research_envelope.decision_points >= 3:
    complexity_boost_flags.append("multi_decision")

# 升级规则
IF len(complexity_boost_flags) >= 2 AND complexity == "small":
    complexity = "medium"  # small 至少升级到 medium

IF len(complexity_boost_flags) >= 3 AND complexity == "medium":
    complexity = "large"   # medium 升级到 large

IF "high_risk_domain" IN complexity_boost_flags AND complexity == "small":
    complexity = "medium"  # 高风险域单独命中即不允许 small
```

| 维度 | 检测方式 | 升级效果 |
|------|---------|---------|
| 跨模块 (`cross_module`) | modified_files 路径的顶层目录去重计数 >= 2 | 参与组合升级 |
| 高风险域 (`high_risk_domain`) | 关键词匹配（安全/认证/支付/性能/迁移/加密） | 单独命中: small→medium; 参与组合升级 |
| 新依赖 (`new_dependency`) | grep 项目依赖清单不存在该依赖 | 参与组合升级 |
| 非功能约束 (`non_functional_constraint`) | 需求含延迟/吞吐/并发/兼容/一致性关键词 | 参与组合升级 |
| 多决策点 (`multi_decision`) | decision_points 数量 >= 3 | 参与组合升级 |

### 分路策略（弹性收敛重构）

| 复杂度 | 清晰度阈值 | 提问策略 | 苏格拉底/挑战代理 |
|--------|-----------|---------|-----------------|
| small | 0.70（宽松） | 合并决策点一次确认 | 禁用苏格拉底；挑战代理按配置 |
| medium | 0.80（标准） | 一次一问（每轮 1 个决策点） | 遵循 config；挑战代理按配置 |
| large | 0.85（严格） | 一次一问 + scope creep 检查 | 强制苏格拉底；挑战代理按配置 |

> **轮数不再硬性限制**。讨论轮数由清晰度评分自然收敛决定（详见 `phase1-clarity-scoring.md`）。
> 安全阀: `soft_warning_rounds`（默认 8）软提醒 + `max_rounds`（默认 15）硬上限。

### 降级处理

如果 Research Agent 被跳过（research_status: "skipped"），默认 complexity = `"medium"`。

## 1.6 多轮决策循环详细流程

### 主动讨论协议

**核心原则**：但凡存在不确定性，主动列出可选方案，由用户决策。

对每个识别到的不确定点，编排器必须：

1. **结构化呈现**决策卡片：
   - 列出 2-4 个可选方案
   - 每个方案标注优点、缺点、影响范围
   - 标记推荐方案（基于调研数据）
   - 说明"如果不决策"的默认行为和风险

2. **分类处理**：
   - **技术决策**（框架选择、架构模式）→ 完整决策卡片 + 调研数据支撑
   - **范围决策**（功能边界、MVP 范围）→ 优先级矩阵 + 工期影响
   - **风险决策**（安全、性能、兼容性）→ 风险评估表 + 缓解方案

3. **等待用户选择**：通过 AskUserQuestion 呈现，绝不自行决策

### Small 复杂度快速路径

当 complexity == "small" 时：
1. 将所有决策点合并为一次 AskUserQuestion（多选模式或逐一快速确认）
2. 展示调研结论 + 推荐方案，让用户一次性确认
3. 清晰度阈值 0.70（宽松），通常 1-2 轮即可达标退出

### Large 复杂度强化路径

当 complexity == "large" 时，强制附加以下检查：
1. 每轮循环后检查是否有需求蔓延（scope creep）迹象
2. 如果功能点持续增加，主动提出 MVP 范围收敛建议（或由简化者挑战代理自动触发）
3. 清晰度阈值 0.85（严格），强制苏格拉底模式 + 挑战代理
4. 一次一问原则：每轮只提 1 个决策点，按最弱清晰度维度优先选择

### hash 计算方法

```python
import hashlib, json

def compute_packet_hash(packet: dict) -> str:
    """计算 requirement-packet.json 的 hash（排除 hash 字段本身）"""
    canonical = {k: v for k, v in sorted(packet.items()) if k != "hash"}
    canonical_json = json.dumps(canonical, sort_keys=True, ensure_ascii=False)
    return hashlib.sha256(canonical_json.encode("utf-8")).hexdigest()
```
