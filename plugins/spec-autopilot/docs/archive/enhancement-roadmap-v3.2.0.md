### P0 — Bug修复：JSON envelope 误匹配

**目标**

修复 Phase 5 实施阶段中，Claude 返回的 JSON envelope 被正则错误匹配的问题。
该 bug 会导致任务状态解析失败，触发不必要的重试，严重时阻塞整个编排流程。

**设计要点**

- 采用两遍搜索策略：第一遍严格匹配标准 envelope 格式 `{"type":"...","payload":...}`；
  第二遍在宽松模式下搜索嵌入在 markdown 代码块中的 JSON 片段。
- 引入 envelope 签名校验：对合法 envelope 的 `type` 字段做白名单过滤，
  排除代码示例、文档引用等非协议 JSON。
- 增加单元测试覆盖：至少 15 个边界用例（嵌套 JSON、转义字符、多行 JSON 等）。

**实现路径**

1. 重构 `envelope-parser.ts` 中的正则表达式，拆分为严格模式和宽松模式。
2. 新增 `EnvelopeValidator` 类，负责 type 白名单校验和结构验证。
3. 在 `phase-5-executor` 中替换现有的 `JSON.parse` 直接调用为 `EnvelopeValidator.parse()`。
4. 补充回归测试套件。

**预期影响**

- 消除因误匹配导致的 Phase 5 中断（生产环境约 5% 的失败率）。
- 提升编排流程的稳定性和可观测性。

### P1 — Phase 5 并行执行（混合模式）

**目标**

在 Phase 5（代码实施）阶段引入混合并行模式，允许无依赖关系的任务组并行执行，
同时保持有依赖关系的任务串行执行，显著缩短大型功能的实施时间。

**设计要点**

- **依赖分析**：解析 `tasks.md` 中的任务依赖关系（`blockedBy` 字段），
  构建有向无环图（DAG）。
- **分组策略**：基于 DAG 进行拓扑排序，将同一层级的无依赖任务归为一组，
  每组内的任务可并行执行。
- **并行派发**：使用 git worktree 为每个并行任务创建独立工作目录，
  通过 Claude Code 的 Task 工具派发子 agent。
- **冲突检测**：并行任务完成后，执行文件级冲突检测；若存在冲突文件，
  自动触发 review agent 进行合并决策。
- **合并策略**：无冲突任务直接 cherry-pick；有冲突任务通过三方合并 +
  人工确认的方式解决。

**实现路径**

1. 实现 `TaskDependencyAnalyzer`：解析 tasks.md → DAG → 分组列表。
2. 实现 `WorktreeManager`：管理 git worktree 的创建、清理和状态追踪。
3. 实现 `ParallelDispatcher`：并行派发任务到各 worktree，收集执行结果。
4. 实现 `MergeCoordinator`：冲突检测 + 合并策略 + review 触发。
5. 重构 `phase-5-executor`：集成上述模块，支持串行/并行/混合三种模式切换。

**预期影响**

- 对于 8+ 任务的大型功能，实施时间预计缩短 40-60%。
- 保持与现有串行模式的完全兼容，可通过配置切换。

### P1 — 通用并行编排模块 parallel-dispatch

**目标**

将 Phase 5 并行执行中的通用能力抽取为独立模块 `parallel-dispatch`，
使其可被 Phase 1、Phase 4、Phase 6 等其他阶段复用。

**设计要点**

- **抽象接口**：定义 `ParallelTask` 接口，包含 `id`、`execute()`、
  `dependencies`、`workdir` 等属性。
- **调度器**：实现基于信号量的并发控制（`maxConcurrency` 可配置），
  支持动态调整并发度。
- **事件总线**：任务生命周期事件（start、progress、complete、error）
  通过 EventEmitter 广播，供 UI 层消费。
- **超时与重试**：每个任务可配置独立的超时时间和重试策略。
- **资源隔离**：支持 worktree 隔离（文件操作类任务）和进程隔离
  （CPU 密集型任务）两种模式。

**实现路径**

1. 创建 `plugins/parallel-dispatch/` 目录，定义核心接口和类型。
2. 实现 `Scheduler` 类：拓扑排序 + 信号量控制 + 事件广播。
3. 实现 `WorktreeIsolator`：基于 git worktree 的资源隔离。
4. 实现 `ProcessIsolator`：基于子进程的资源隔离。
5. 编写适配层，将 Phase 5 的现有逻辑迁移到 parallel-dispatch。
6. 单元测试 + 集成测试覆盖。

**预期影响**

- 为所有需要并行的阶段提供统一的基础设施。
- 降低各阶段实现并行的开发成本（预计减少 60% 的重复代码）。

### P2 — 测试报告 Allure 统一集成

**目标**

将 autopilot 流程中的所有测试产出（API 测试、UI 测试、E2E 测试）
统一集成到 Allure 报告框架，提供一站式的测试可视化和趋势分析。

**设计要点**

- **报告收集**：在 Phase 6 结束后，自动扫描各测试框架的输出目录，
  收集 pytest 的 allure-results、Playwright 的 JSON 报告等。
- **格式转换**：实现 Playwright JSON → Allure XML 的转换器，
  确保所有测试结果格式统一。
- **报告生成**：调用 `allure generate` 生成统一的 HTML 报告。
- **趋势追踪**：保留历史报告数据，支持跨迭代的测试趋势分析。
- **集成入口**：在 Phase 7 归档时，将 Allure 报告链接写入归档摘要。

**实现路径**

1. 在 `plugins/` 下创建 `allure-integration/` 模块。
2. 实现 `ReportCollector`：扫描并收集各框架测试结果。
3. 实现 `PlaywrightToAllureConverter`：格式转换。
4. 实现 `AllureReportGenerator`：调用 allure CLI 生成报告。
5. 实现 `TrendTracker`：历史数据存储和趋势计算。
6. 集成到 Phase 6 和 Phase 7 的执行流程中。

**预期影响**

- 提供统一的测试结果视图，减少手动查看多个报告的时间。
- 支持测试质量趋势分析，辅助质量决策。

### P2 — Phase 4 需求驱动测试用例生成

**目标**

增强 Phase 4（测试设计）的自动化能力，使其能够直接从 Phase 1 的需求文档
和 Phase 3 的设计文档中提取测试场景，自动生成测试用例框架。

**设计要点**

- **需求解析**：从 proposal.md 和 design.md 中提取功能点、边界条件、
  异常场景等测试维度。
- **用例模板**：预定义 API 测试、UI 测试、E2E 测试的用例模板，
  根据功能类型自动选择合适的模板。
- **覆盖率矩阵**：生成需求-用例追溯矩阵，确保每个需求点都有对应的测试覆盖。
- **智能补全**：对于常见的 CRUD 操作、表单验证、权限控制等模式，
  自动生成标准化的测试用例集。

**实现路径**

1. 实现 `RequirementExtractor`：从 markdown 文档中提取结构化需求。
2. 实现 `TestScenarioGenerator`：基于需求生成测试场景列表。
3. 实现 `TestCaseTemplateEngine`：将测试场景填充到用例模板中。
4. 实现 `CoverageMatrixBuilder`：生成需求-用例追溯矩阵。
5. 集成到 Phase 4 执行流程，替换现有的纯手动用例设计。

**预期影响**

- Phase 4 的用例设计时间预计缩短 50%。
- 测试覆盖率提升至 90%+（当前约 70%）。
- 减少人工遗漏关键测试场景的风险。

### P3 — Phase 1/4/6 并行复用

**目标**

将 parallel-dispatch 模块复用到 Phase 1（需求分析）、Phase 4（测试设计）、
Phase 6（测试执行）中，在各阶段内部实现细粒度的并行处理。

**设计要点**

- **Phase 1 并行点**：
  - 多个需求维度的分析可并行（功能分析、技术调研、竞品对比）。
  - 汇合策略：所有分析完成后，由主 agent 综合产出 proposal。
- **Phase 4 并行点**：
  - 不同测试类型的用例设计可并行（API 测试、UI 测试、E2E 测试）。
  - 汇合策略：各类型用例独立生成，最终合并到统一的 test-specs 目录。
- **Phase 6 并行点**：
  - 不同测试套件可并行执行（前提是测试间无共享状态）。
  - 汇合策略：收集所有测试结果，统一生成报告。

**实现路径**

1. 为 Phase 1 定义 `AnalysisTask` 类型，实现 `ParallelTask` 接口。
2. 为 Phase 4 定义 `TestDesignTask` 类型，实现 `ParallelTask` 接口。
3. 为 Phase 6 定义 `TestExecutionTask` 类型，实现 `ParallelTask` 接口。
4. 在各 Phase 的 executor 中集成 `Scheduler` 调用。
5. 配置各 Phase 的并发度和隔离策略。

**预期影响**

- Phase 1 分析时间缩短约 30%。
- Phase 4 设计时间缩短约 40%。
- Phase 6 测试执行时间缩短约 50%（取决于测试套件的独立性）。

### P3 — 代码生成约束增强

**目标**

增强 Phase 5 代码生成过程中的约束检查能力，减少生成代码的质量问题，
降低 review 和返工的成本。

**设计要点**

- **预检查门禁**：在代码生成前，检查目标文件的现有结构、导入规范、
  命名约定等，将约束条件注入到 prompt 中。
- **增量验证**：每个任务完成后立即执行类型检查、lint 检查和构建验证，
  发现问题立即修复，避免问题累积。
- **模式匹配**：维护常见代码模式库（如 Vue 组件结构、Spring Controller 结构），
  生成代码时强制匹配已有模式。
- **文件大小监控**：实时监控生成文件的行数，超过阈值自动触发拆分建议。

**实现路径**

1. 实现 `CodeConstraintChecker`：预检查目标文件的约束条件。
2. 实现 `IncrementalValidator`：增量式的代码质量验证。
3. 实现 `PatternMatcher`：代码模式库和匹配引擎。
4. 实现 `FileSizeMonitor`：文件大小监控和拆分建议。
5. 集成到 Phase 5 的任务执行流程中。

**预期影响**

- 代码生成的一次通过率从 70% 提升至 90%。
- 减少 50% 的 review 返工。
- 保持代码库的一致性和可维护性。

### P4 — Phase 1 需求理解增强（联网调研并行）

**目标**

增强 Phase 1 的需求理解能力，支持在需求分析过程中并行进行联网调研，
获取相关技术方案、竞品分析、最佳实践等外部信息。

**设计要点**

- **调研触发**：当需求涉及未知技术栈、第三方集成、性能基准等场景时，
  自动触发联网调研任务。
- **并行执行**：调研任务与本地需求分析并行执行，互不阻塞。
- **信息整合**：调研结果自动整理为结构化摘要，注入到 proposal 的
  "技术调研" 章节。
- **缓存机制**：对于相同或相似的调研主题，复用历史调研结果，
  减少重复的网络请求。

**实现路径**

1. 实现 `ResearchTrigger`：基于需求关键词判断是否需要联网调研。
2. 实现 `WebResearchAgent`：封装搜索 API 和内容提取逻辑。
3. 实现 `ResearchCache`：调研结果的本地缓存和检索。
4. 实现 `ResearchIntegrator`：将调研结果整合到 proposal 文档中。
5. 通过 parallel-dispatch 实现调研与分析的并行。

**预期影响**

- 提升需求分析的全面性，特别是涉及新技术的场景。
- 减少后续阶段因技术方案不当导致的返工。
- 调研结果可复用，长期降低分析成本。

### P4 — 主动决策增强（所有复杂度）

**目标**

增强 autopilot 在各阶段的主动决策能力，减少对用户确认的依赖，
使得简单到中等复杂度的功能可以完全自动化完成。

**设计要点**

- **决策分级**：将决策分为三级——自动决策（低风险）、建议决策（中风险）、
  人工决策（高风险）。
- **风险评估**：基于变更范围、文件影响数、是否涉及核心模块等维度，
  自动评估决策风险等级。
- **决策日志**：所有自动决策记录到 `decisions.log`，支持事后审计和复盘。
- **回滚支持**：每个自动决策关联一个回滚操作，支持一键撤销。

**实现路径**

1. 定义决策分级模型和风险评估维度。
2. 实现 `RiskAssessor`：自动评估决策风险等级。
3. 实现 `DecisionEngine`：根据风险等级自动执行或请求确认。
4. 实现 `DecisionLogger`：决策日志记录和查询。
5. 实现 `RollbackManager`：决策回滚支持。
6. 集成到各 Phase 的执行流程中。

**预期影响**

- 简单功能的人工干预降低至 0 次。
- 中等复杂度功能的人工干预从 5-8 次降低至 1-2 次。
- 决策可追溯，提升流程透明度。

### P5 — 持续学习系统（Instincts）

**目标**

构建 autopilot 的持续学习系统，使其能够从历史项目、用户反馈、
失败案例中学习，不断提升自动化的准确性和效率。

**设计要点**

- **经验库**：结构化存储历史项目的成功模式、失败案例和用户偏好。
- **模式识别**：从经验库中识别重复出现的模式，生成"直觉规则"（Instincts）。
- **规则应用**：在各阶段的决策和生成过程中，自动检索和应用相关的直觉规则。
- **反馈回路**：用户对 autopilot 产出的修改自动反馈到经验库，
  持续优化直觉规则。
- **规则治理**：支持查看、编辑、禁用直觉规则，避免错误规则的传播。

**实现路径**

1. 设计经验库的数据模型和存储格式（基于本地文件系统）。
2. 实现 `ExperienceCollector`：从项目历史中收集经验数据。
3. 实现 `PatternRecognizer`：从经验数据中提取直觉规则。
4. 实现 `InstinctEngine`：直觉规则的存储、检索和应用。
5. 实现 `FeedbackLoop`：用户反馈的自动收集和规则更新。
6. 实现 `InstinctGovernance`：规则的查看、编辑和管理界面。

**预期影响**

- 随着使用量增加，autopilot 的决策准确率持续提升。
- 减少重复犯错，特别是项目特定的约束和偏好。
- 形成团队级的知识沉淀，新成员可快速受益。

### Agent Teams 集成

**描述**

当 Claude Code 原生支持 Agent Teams 功能后，将 autopilot 的并行执行架构
从手动 git worktree + Task 工具迁移到原生 Agent Teams API。这将带来：

- 更稳定的并行执行（原生支持 vs 手动编排）。
- 更好的资源管理（原生调度 vs 信号量控制）。
- 更丰富的协作模式（agent 间通信、共享上下文等）。

迁移策略：保留 parallel-dispatch 的抽象接口，仅替换底层实现，
确保上层逻辑无感知切换。

---

### Smart Model Routing

**描述**

实现自动化的模型路由策略，根据任务复杂度自动选择最合适的模型：

- **Haiku**：简单的文件读取、格式转换、模板填充等低复杂度任务。
- **Sonnet**：标准的代码生成、测试编写、文档生成等中复杂度任务。
- **Opus**：架构设计、复杂调试、跨模块重构等高复杂度任务。

预期可节省 30-50% 的 token 消耗，同时保持产出质量。

路由决策基于：任务类型标签、历史 token 消耗、产出质量评分等多维度。

---

### TDD 可选模式

**描述**

借鉴 Superpowers 框架的 TDD 实践，在 Phase 5 中提供可选的
RED-GREEN-REFACTOR 循环模式：

- **RED**：先编写失败的测试用例，明确预期行为。
- **GREEN**：编写最少的代码使测试通过。
- **REFACTOR**：在测试保护下重构代码，提升质量。

适用于对代码质量要求较高的核心模块开发。通过配置项控制是否启用。

---

### Document Sharding

**描述**

借鉴 BMAD 框架的文档分片技术，对大型制品（如 design.md、specs/）
自动进行智能拆分：

- 基于内容语义自动识别拆分点。
- 生成目录索引文件，保持整体可导航性。
- 各分片独立加载，减少单次 context 占用。
- 支持按需加载和懒加载模式。

解决当前大型功能的文档超过 1500 行限制的问题。

---

### 跨平台支持

**描述**

将 autopilot 的编排能力适配到更多 AI 编码工具：

- **Cursor**：适配 Cursor 的 Agent 模式和 Composer API。
- **OpenCode**：适配 OpenCode 的插件系统。
- **Codex**：适配 OpenAI Codex 的代码生成 API。

通过抽象适配层实现，核心编排逻辑保持不变，仅替换底层的
agent 调用和工具调用接口。
