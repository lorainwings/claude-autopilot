## 1. 变更概述

### 1.1 迭代目标

v3.2.0 是 autopilot 的重要性能与质量提升版本，核心目标：

1. **并行化**：引入 Phase 5 混合并行模式，将大型功能的实施时间缩短 40-60%。
2. **模块化**：抽取通用并行编排模块 `parallel-dispatch`，为后续阶段并行提供基础。
3. **质量提升**：统一 Allure 测试报告、增强代码生成约束、优化需求驱动测试。
4. **稳定性**：修复 JSON envelope 误匹配 bug，消除 5% 的随机失败。
5. **智能化**：增强需求理解和主动决策能力，减少人工干预。

### 1.2 版本范围

| 模块 | 变更类型 | 影响面 |
|------|----------|--------|
| parallel-dispatch | 新增 | 核心基础设施 |
| phase-5-executor | 重构 | 实施阶段 |
| phase-1-executor | 增强 | 需求分析阶段 |
| phase-4-executor | 增强 | 测试设计阶段 |
| phase-6-executor | 增强 | 测试执行阶段 |
| envelope-parser | 修复 | 消息解析 |
| allure-integration | 新增 | 测试报告 |
| decision-engine | 新增 | 决策系统 |

### 1.3 非目标

- 不涉及 Phase 2（OpenSpec 制品生成）和 Phase 3（设计）的变更。
- 不涉及 Agent Teams 集成（v4.0.0 目标）。
- 不涉及跨平台支持。

## 2. 架构变更

### 2.1 parallel-dispatch 模块设计

#### 2.1.1 模块定位

`parallel-dispatch` 是 autopilot 的核心基础设施模块，提供通用的
并行任务编排能力。它位于编排层（Orchestration Layer）和执行层
（Execution Layer）之间。

```
┌─────────────────────────────────────────────┐
│              Orchestration Layer             │
│  (phase-1, phase-4, phase-5, phase-6)       │
├─────────────────────────────────────────────┤
│           parallel-dispatch                  │
│  ┌──────────┬──────────┬──────────────────┐ │
│  │ Scheduler│ Isolator │ EventBus         │ │
│  │          │          │                  │ │
│  │ - DAG    │ - Worktree│ - task.start    │ │
│  │ - Topo   │ - Process │ - task.progress │ │
│  │ - Sema   │           │ - task.complete │ │
│  └──────────┴──────────┴──────────────────┘ │
├─────────────────────────────────────────────┤
│              Execution Layer                 │
│  (Claude Task, git worktree, shell)          │
└─────────────────────────────────────────────┘
```

#### 2.1.2 核心接口

```typescript
// 并行任务接口
interface ParallelTask {
  id: string;
  name: string;
  dependencies: string[];       // 依赖的任务 ID
  isolationMode: 'worktree' | 'process' | 'none';
  timeout?: number;             // 超时时间（ms）
  retryPolicy?: RetryPolicy;
  execute(context: TaskContext): Promise<TaskResult>;
}

// 调度器配置
interface SchedulerConfig {
  maxConcurrency: number;       // 最大并发数
  strategy: 'eager' | 'conservative';
  onConflict: 'fail' | 'merge' | 'manual';
}

// 任务结果
interface TaskResult {
  status: 'success' | 'failure' | 'skipped';
  output?: unknown;
  changedFiles?: string[];
  duration: number;
  error?: Error;
}

// 重试策略
interface RetryPolicy {
  maxRetries: number;
  backoffMs: number;
  retryableErrors?: string[];
}
```

#### 2.1.3 调度算法

调度器采用基于拓扑排序的层级调度算法：

1. 解析任务依赖关系，构建 DAG。
2. 对 DAG 执行拓扑排序，得到层级列表。
3. 同一层级内的任务无依赖关系，可并行执行。
4. 使用信号量控制最大并发数。
5. 当一个层级的所有任务完成后，开始下一层级。

```
Layer 0: [Task-A, Task-B]  ── 并行执行
                ↓
Layer 1: [Task-C]           ── 等待 A、B 完成后执行
                ↓
Layer 2: [Task-D, Task-E]  ── 并行执行
                ↓
Layer 3: [Task-F]           ── 等待 D、E 完成后执行
```

#### 2.1.4 事件系统

```typescript
// 事件类型
type TaskEvent =
  | { type: 'task.queued'; taskId: string }
  | { type: 'task.started'; taskId: string; workdir: string }
  | { type: 'task.progress'; taskId: string; message: string }
  | { type: 'task.completed'; taskId: string; result: TaskResult }
  | { type: 'task.failed'; taskId: string; error: Error }
  | { type: 'layer.started'; layerIndex: number; tasks: string[] }
  | { type: 'layer.completed'; layerIndex: number }
  | { type: 'schedule.completed'; results: Map<string, TaskResult> };
```

### 3.2 依赖分析

依赖分析器从 `tasks.md` 中提取任务定义和依赖关系。

**输入格式**（tasks.md 示例）：

```markdown
## Tasks

### Task 1: 创建数据模型
- blockedBy: []
- files: src/models/user.ts

### Task 2: 实现 API 路由
- blockedBy: [Task 1]
- files: src/routes/user.ts

### Task 3: 编写单元测试
- blockedBy: [Task 1]
- files: tests/user.test.ts

### Task 4: 集成测试
- blockedBy: [Task 2, Task 3]
- files: tests/integration/user.test.ts
```

**输出**（DAG + 层级分组）：

```
DAG: T1 → T2, T1 → T3, T2 → T4, T3 → T4

Layer 0: [T1]         串行（单任务）
Layer 1: [T2, T3]     并行（无互相依赖）
Layer 2: [T4]         串行（单任务）
```

### 3.3 Worktree 管理

每个并行任务在独立的 git worktree 中执行，确保文件系统级别的隔离。

```typescript
class WorktreeManager {
  private readonly baseDir: string;
  private readonly worktrees: Map<string, WorktreeInfo> = new Map();

  // 创建 worktree
  async create(taskId: string, branch: string): Promise<string> {
    const worktreeDir = path.join(this.baseDir, `.worktree-${taskId}`);
    await exec(`git worktree add ${worktreeDir} -b task/${taskId} ${branch}`);
    this.worktrees.set(taskId, { dir: worktreeDir, branch: `task/${taskId}` });
    return worktreeDir;
  }

  // 清理 worktree
  async cleanup(taskId: string): Promise<void> {
    const info = this.worktrees.get(taskId);
    if (info) {
      await exec(`git worktree remove ${info.dir} --force`);
      await exec(`git branch -D ${info.branch}`);
      this.worktrees.delete(taskId);
    }
  }

  // 清理所有 worktree
  async cleanupAll(): Promise<void> {
    for (const taskId of this.worktrees.keys()) {
      await this.cleanup(taskId);
    }
  }
}
```

### 3.4 冲突检测与合并

并行任务完成后，需要检测各 worktree 的变更文件是否存在冲突。

**冲突检测策略**：

1. 收集每个 worktree 的变更文件列表（`git diff --name-only`）。
2. 计算文件交集——若两个任务修改了同一文件，标记为潜在冲突。
3. 对潜在冲突文件执行内容级别的冲突检测（`git merge --no-commit`）。

**合并策略**：

| 场景 | 策略 |
|------|------|
| 无文件重叠 | 直接 cherry-pick 各任务的 commit |
| 文件重叠但无内容冲突 | 自动三方合并 |
| 存在内容冲突 | 触发 review agent，由 AI 辅助解决 |
| review agent 无法解决 | 回退到串行模式重新执行 |

### 3.5 模式切换

Phase 5 支持三种执行模式：

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| serial | 传统串行执行 | 高度耦合的小型变更 |
| parallel | 全并行执行 | 完全独立的任务集 |
| hybrid | 混合模式（默认） | 一般场景 |

通过配置项控制：

```yaml
# .autopilot/config.yaml
phase5:
  execution_mode: hybrid    # serial | parallel | hybrid
  max_concurrency: 3        # 最大并发数
  conflict_strategy: merge  # fail | merge | manual
```

### 4.1 Phase 1 并行点

Phase 1（需求分析）中可并行的子任务：

```
┌───────────────────────────────────────────┐
│  Phase 1: 需求分析                        │
│                                           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │功能分析   │ │技术调研   │ │竞品对比   │  │

│                    ▼                      │
│           ┌──────────────┐                │

│           │ 生成 proposal │                │
```

**汇合策略**：

- 所有并行子任务完成后，主 agent 读取各子任务的输出。

**隔离模式**：`none`（无文件系统变更，仅产出文本）。

```
│  ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │API 测试   │ │UI 测试    │ │E2E 测试   │  │
│  │用例设计   │ │用例设计   │ │用例设计   │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘  │
│       │            │            │         │
│       ▼            ▼            ▼         │
│  test-specs/   test-specs/  test-specs/   │
│  api/          ui/          e2e/          │
└───────────────────────────────────────────┘
```

**汇合策略**：

- 各测试类型的用例独立生成到各自的子目录。
- 无需合并步骤，各子目录天然隔离。
- 完成后统一执行 `test_counts` 检查（≥5 个用例/类型）。

**隔离模式**：`none`（各写各的目录，无冲突风险）。

Phase 6（测试执行）中可并行的子任务：

```
┌───────────────────────────────────────────┐

│                                           │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐  │

│                    ▼                      │
│           ┌──────────────┐                │
│           │ 结果汇总      │                │
│           │ 报告生成      │                │
```

**汇合策略**：

- 收集所有测试框架的输出文件（allure-results、playwright-report）。
- 统一生成 Allure 报告（参见第 5 节）。

**隔离模式**：`process`（各测试进程独立运行）。

**并发限制**：

- 默认 `maxConcurrency: 2`（避免 CI 资源争用）。
- Playwright 测试使用 `--workers=1` 避免浏览器资源冲突。

## 6. 能力增强设计

### 6.1 代码生成约束增强

#### 6.1.1 预检查门禁

在代码生成前，`CodeConstraintChecker` 执行以下检查：

```typescript
interface ConstraintCheckResult {
  passed: boolean;
  constraints: Constraint[];  // 需要注入到 prompt 的约束
  warnings: string[];         // 警告信息
}

interface Constraint {
  type: 'naming' | 'import' | 'structure' | 'size' | 'pattern';
  description: string;
  example?: string;
}
```

**检查项**：

| 检查类型 | 检查内容 | 注入方式 |
|----------|----------|----------|
| 命名约定 | 文件名/类名/变量名风格 | prompt 前置约束 |
| 导入规范 | 导入路径别名、排序规则 | prompt 前置约束 |
| 文件结构 | 组件/类的代码结构模板 | prompt 示例代码 |
| 文件大小 | 目标文件现有行数 + 预估增量 | prompt 警告 |
| 代码模式 | 已有代码的模式匹配结果 | prompt 参考代码 |

```typescript
enum RiskLevel {
  LOW = 'low',       // 自动决策
  MEDIUM = 'medium', // 建议决策（显示建议，自动执行，可撤销）
```

#### 6.4.2 风险评估维度

| 维度 | 权重 | 低风险（0-0.3） | 中风险（0.3-0.7） | 高风险（0.7-1.0） |
|------|------|-----------------|-------------------|-------------------|
| 变更范围 | 0.3 | ≤3 文件 | 4-10 文件 | >10 文件 |
| 核心模块 | 0.3 | 非核心/测试 | 业务模块 | 基础设施/配置 |
| 破坏性变更 | 0.2 | 纯新增 | 修改已有接口 | 删除/重命名 |
| 数据影响 | 0.2 | 无数据变更 | 数据格式变更 | 数据迁移/删除 |

**综合风险分数** = Σ(维度分数 × 权重)

- 分数 < 0.3 → LOW（自动决策）
- 0.3 ≤ 分数 < 0.7 → MEDIUM（建议决策）
- 分数 ≥ 0.7 → HIGH（人工决策）

## 7. JSON envelope 修复

### 7.1 问题描述

Phase 5 实施阶段中，Claude 返回的消息有时包含 JSON 代码示例或文档引用，
这些 JSON 片段被现有的正则表达式错误匹配为协议 envelope，导致：

- 状态解析失败
- 触发不必要的重试
- 严重时阻塞整个编排流程

生产环境中约 5% 的 Phase 5 执行会受此 bug 影响。

### 7.2 两遍搜索策略

**第一遍：严格匹配**

```typescript
// 严格模式正则：匹配标准 envelope 格式
const STRICT_PATTERN = /^{"type":"(task_status|task_result|phase_gate)","payload":\{.*\}}$/m;

function strictMatch(text: string): EnvelopeMatch | null {
  const match = text.match(STRICT_PATTERN);
  if (match) {
    try {
      const parsed = JSON.parse(match[0]);
      if (isValidEnvelope(parsed)) {
        return { envelope: parsed, confidence: 'high' };
      }
    } catch {
      // 非法 JSON，跳过
    }
  }
  return null;
}
```

**第二遍：宽松匹配**

```typescript
// 宽松模式：搜索嵌入在 markdown 中的 JSON
const LOOSE_PATTERN = /```(?:json)?\s*\n(\{[\s\S]*?"type"\s*:\s*"(?:task_status|task_result|phase_gate)"[\s\S]*?\})\s*\n```/g;

function looseMatch(text: string): EnvelopeMatch | null {
  const matches = [...text.matchAll(LOOSE_PATTERN)];
  for (const match of matches) {
    try {
      const parsed = JSON.parse(match[1]);
      if (isValidEnvelope(parsed)) {
        return { envelope: parsed, confidence: 'medium' };
      }
    } catch {
      continue;
    }
  }
  return null;
}
```

### 7.3 Envelope 验证器

```typescript
// type 白名单
const VALID_ENVELOPE_TYPES = new Set([
  'task_status',
  'task_result',
  'phase_gate',
  'error',
  'heartbeat',
]);

function isValidEnvelope(obj: unknown): obj is Envelope {
  if (typeof obj !== 'object' || obj === null) return false;
  const record = obj as Record<string, unknown>;

  // 必须有 type 字段且在白名单中
  if (!VALID_ENVELOPE_TYPES.has(record.type as string)) return false;

  // 必须有 payload 字段且为对象
  if (typeof record.payload !== 'object' || record.payload === null) return false;

  // 不能有多余的顶层字段（排除代码示例中的扩展 JSON）
  const allowedKeys = new Set(['type', 'payload', 'timestamp', 'taskId']);
  const extraKeys = Object.keys(record).filter(k => !allowedKeys.has(k));
  if (extraKeys.length > 0) return false;

  return true;
}
```

### 7.4 测试覆盖

至少 15 个边界用例：

| # | 用例描述 | 预期结果 |
|---|----------|----------|
| 1 | 标准 envelope | 严格匹配成功 |
| 2 | markdown 代码块中的 envelope | 宽松匹配成功 |
| 3 | 非协议 JSON 对象 | 不匹配 |
| 4 | 嵌套 JSON（内层包含 type 字段） | 不误匹配 |
| 5 | 转义字符在 JSON 中 | 正确解析 |
| 6 | 多行 JSON | 正确匹配 |

| 8 | 空 payload | 匹配但标记为异常 |
| 9 | type 字段为空字符串 | 不匹配 |
| 10 | 非白名单 type 值 | 不匹配 |
| 11 | payload 为数组而非对象 | 不匹配 |
| 12 | JSON 在普通文本中（非代码块） | 宽松匹配 |
| 13 | 损坏的 JSON（缺少闭合括号） | 不匹配，不抛异常 |
| 14 | Unicode 字符在 JSON 值中 | 正确解析 |
| 15 | 超大 JSON（>10KB） | 超时保护，不匹配 |
