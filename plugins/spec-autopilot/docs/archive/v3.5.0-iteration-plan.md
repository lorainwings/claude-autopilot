## 3. 修复 .autopilot-active 写入问题

### 3.1 Root Cause

1. **Write 未先 Read**：Write 工具要求对已有文件先 Read，Phase 0 Step 9 直接 Write 导致首次报错
2. **"Update" 非工具**：Step 10 使用 `Update(...)` 应改为 `Edit(...)`
3. **目录可能不存在**：首次运行时 `openspec/changes/` 目录未必存在

### 3.2 Fix（集成在 autopilot-lockfile skill 中）

锁文件的 Read→Write→Verify 全部由后台 Agent 封装，从根本上解决：
- 后台 Agent 内部自行 Read 后 Write（满足工具前置要求）
- 后台 Agent 先 `mkdir -p` 确保目录存在
- 后台 Agent 用 Edit 更新字段（满足 Edit 工具前置要求：先 Read）
- 主线程不再直接操作锁文件（除 Phase 7 删除用 Bash rm）

## 4. 后台 Agent 管理锁文件

### 4.1 autopilot-lockfile/SKILL.md 结构

```markdown
# 操作类型

## 操作 1: 创建锁文件 (Phase 0 Step 9)
Agent(run_in_background: true, prompt: "
  <!-- lockfile-writer -->
  1. mkdir -p ${session_cwd}/openspec/changes/
  2. test -f → exists: Read → PID 冲突检测; new: 跳过
  3. Write(绝对路径, JSON)
  4. Read 验证
  返回: {status, action: created|overwritten|conflict}
")

## 操作 2: 更新 anchor_sha (Phase 0 Step 10)
Agent(run_in_background: true, prompt: "
  <!-- lockfile-updater -->
  1. Read 锁文件
  2. Edit: 替换 anchor_sha 字段
  3. Read 验证
  返回: {status, anchor_sha}
")

## 操作 3: 删除锁文件 (Phase 7 Step 7)
Bash("rm -f ${session_cwd}/openspec/changes/.autopilot-active")
— 删除操作直接用 Bash，无需 Agent 封装

## 冲突处理
- conflict → 主线程 AskUserQuestion 让用户决定覆盖/中止
```

### 4.2 主线程集成点

| 阶段 | 操作 | 调用方式 |
|------|------|---------|
| Phase 0 Step 9 | 创建锁文件 | autopilot-phase0 调用 lockfile 操作 1 |
| Phase 0 Step 10 | 更新 anchor_sha | autopilot-phase0 调用 lockfile 操作 2 |
| Phase 7 Step 7 | 删除锁文件 | autopilot-phase7 直接 Bash rm |
