# Phase 1 需求质量 Benchmark 报告

> 评审日期: 2026-03-13
> 评审范围: `plugins/spec-autopilot` 插件 Phase 1（需求分析阶段）
> 评审版本: v4.0.0-wave4 (commit 08d346f)

---

## 1. 执行摘要

Phase 1 是 autopilot 流水线的需求理解与多轮决策阶段，承担着将模糊用户意图转化为可执行技术规格的关键任务。本报告通过三个不同维度的需求输入场景，对 Phase 1 的 10 步流程进行系统性评审。

**总体评价**: Phase 1 的设计在工程规范性上达到了较高水平，尤其在并行调研架构、结构化决策协议和复杂度路由方面表现突出。主要短板集中在：(1) 模糊需求场景下的主动澄清深度不足；(2) 存量代码调研的静态分析覆盖有盲区；(3) 复杂度路由阈值过度依赖文件数量这一单一维度。

**综合得分: 7.4 / 10**

### 2.1 评审框架

采用"场景驱动 + 指标矩阵"双轨评审法：

2. **指标矩阵**: 从 6 个核心维度（需求理解深度、隐藏约束挖掘、存量代码调研覆盖率、澄清问题工程价值、复杂度路由准确性、多轮对话收敛性）进行量化评分。

## 3. 场景 A 评审: 模糊需求处理能力

### 3.1 输入设计

**需求输入**: "优化性能"

这是一个典型的模糊需求 -- 没有指定优化目标（前端/后端/数据库）、没有量化指标（响应时间目标、吞吐量要求）、没有范围界定（全局还是特定接口）、没有约束条件（预算、时间窗口、兼容性要求）。

**评估重点**: Phase 1 能否通过其流程将"优化性能"这种宽泛需求逐步收敛到可执行的技术方案。

### 3.2 Phase 1 流程分析

**Step 1.1 获取需求来源**: 直接将 "优化性能" 作为 `RAW_REQUIREMENT`。流程设计正确 -- 文本需求直接传入，无需额外处理。但此处缺少一个**前置模糊度检测**环节：当需求文本长度 < 10 字符或不包含任何技术名词时，应立即触发澄清而非直接进入调研。

**Step 1.2 Auto-Scan**: 并行发起三路调研。对"优化性能"这类模糊需求：
- Auto-Scan 将正常扫描项目结构，产出 `project-context.md`、`existing-patterns.md`、`tech-constraints.md`。这一步设计合理 -- 即使需求模糊，项目上下文信息仍然有价值。
- 技术调研 Agent 收到的是 "优化性能" -- 研究范围过于宽泛，可能产出大量不相关信息。Agent prompt 模板中没有针对模糊需求的引导策略。
- 联网搜索的搜索关键词构造规则（`"{技术栈} + {功能关键词} + best practices {当前年份}"`）会生成 "React performance best practices 2026" 之类的泛化查询，ROI 低。

**Step 1.3 技术调研**: Research Agent 的 7 步调研任务中，"相关代码影响分析" 和 "依赖兼容性检查" 在模糊需求下无法有效执行 -- 不知道优化什么，就无法确定影响范围。协议中的"自主方案探索"（v3.0）是一个亮点，但缺乏对需求模糊度的感知机制。

**Step 1.4 复杂度评估**: 基于 `impact_analysis.total_files` 计算复杂度。对"优化性能"，Research Agent 可能返回大量 `modified_files`（因为"性能优化"可能涉及全局），导致被路由到 `large` 复杂度，触发强制苏格拉底模式 -- 这恰好是正确的行为，但是"意外正确"而非"设计正确"。

**Step 1.5 需求分析（business-analyst）**: Agent prompt 注入了完整上下文，但对模糊需求没有特殊处理。理想情况下，business-analyst 应首先输出"需求模糊度评估"，再决定分析策略。

**Step 1.5.5 决策协议**: 结构化决策卡片是本流程的亮点。对"优化性能"，理论上应产出以下决策点：
- 优化目标（前端渲染 / API 响应时间 / 数据库查询 / 构建速度）
- 量化指标（响应时间从 X 降到 Y）
- 优化范围（全局 / 特定页面 / 特定接口）

但这些决策点的产出完全依赖 business-analyst 子 Agent 的能力，Phase 1 流程本身没有针对模糊需求的**强制澄清模板**。

**Step 1.6 多轮决策 LOOP**: 苏格拉底模式的 6 步提问流程（挑战假设、探索替代方案、识别隐含需求、强制排优、魔鬼代言人、最小可行范围）是处理模糊需求的有效工具。然而，它的触发依赖复杂度判定为 `large` 或配置为 `socratic` 模式 -- 如果复杂度意外判定为 `small`（例如 Research Agent 返回较少的文件影响），苏格拉底模式不会激活，模糊需求可能以 1 轮快速确认草草结束。

### 3.3 评分与发现

**得分: 6.5 / 10**

| 维度 | 发现 | 评价 |
|------|------|------|
| 模糊度感知 | 流程中没有显式的"需求模糊度检测"环节，模糊需求与清晰需求走完全相同的流程 | 缺陷 |
| 调研效率 | 三路并行调研对模糊需求可能产出大量噪音信息，浪费 token | 风险 |
| 苏格拉底模式 | 设计精良，6 步提问覆盖了模糊需求澄清的关键维度 | 优势 |
| 强制澄清 | 没有"最低信息量门槛"——理论上可以在用户只说了 "优化性能" 的情况下走完全流程并开始实施 | 缺陷 |
| 决策卡片质量 | 依赖 AI Agent 的理解力，流程层面无法保证产出高质量决策点 | 风险 |

**关键缺陷 A-1**: 缺少前置需求模糊度检测与强制澄清门槛。

**关键缺陷 A-2**: 复杂度路由与苏格拉底模式的联动存在间隙 -- 模糊需求不一定被路由到 `large`，可能跳过深度讨论。

## 4. 场景 B 评审: 跨模块需求分析能力

### 4.1 输入设计

**需求输入**: "添加用户认证系统，支持邮箱/密码登录、OAuth2.0 第三方登录（Google、GitHub）、JWT token 管理、RBAC 权限控制"

这是一个跨多模块的复杂需求：涉及后端（API + 数据库）、前端（登录页 + 权限 UI）、基础设施（JWT 密钥管理 + OAuth 回调配置）、安全（加密 + 令牌刷新策略）。

**评估重点**: Phase 1 能否系统性地分析跨模块依赖、识别隐含的技术约束和集成风险。

### 4.2 Phase 1 流程分析

**Step 1.1 获取需求来源**: 需求文本清晰，直接作为 `RAW_REQUIREMENT`。此场景适合 Phase 1 设计的主路径。

**Step 1.2 Auto-Scan 并行调研**:
- Auto-Scan 将扫描项目结构，识别 `backend/`、`frontend/` 等模块。扫描范围表（技术栈、目录结构、现有模式、依赖清单、项目约束、测试基础设施）覆盖面广。特别是"现有模式"扫描通过 Grep 搜索路由定义、API 端点、认证中间件 -- 这对识别项目是否已有部分认证功能非常关键。
- 技术调研 Agent 将执行 7 步调研，其中"依赖兼容性检查"和"风险识别"对认证系统特别重要。Agent 的"自主方案探索"可能推荐 Passport.js / Spring Security / NextAuth 等成熟方案。
- 联网搜索将因 `security_related: true`（需求含 "auth/JWT"）和 `new_feature: true` 触发**强制搜索**，这是搜索决策规则引擎的正确行为。搜索查询如 "JWT + OAuth2.0 + RBAC best practices 2026" 价值明确。

**Step 1.3 技术调研深度**:
- 对此需求，`complexity` 几乎必定为 `large`（影响 > 5 个文件 + `security_related`），触发 `deep` 调研深度，执行全部 7 项任务，包括"同类实现对比"和"依赖深度分析"。
- 依赖评估（npm 下载量、CVE 检查、许可证兼容性）对选择 OAuth 库至关重要。
- 协议中的搜索结果信任机制（标注来源 + 交叉验证 + 冲突解决）是一个工程亮点。

**Step 1.4 复杂度评估**:
- `total_files` 很可能 > 5，加上 `security_related` 作为升级因子，将被路由到 `large`。
- 复杂度路由的额外调整因子（feasibility.score == "low"、高风险、3+ 新依赖）对此场景可能部分触发。
- 认证系统通常需要引入多个新依赖（JWT 库、OAuth 库、密码加密库），3+ 新依赖条件大概率满足。

**Step 1.5 需求分析**:
- business-analyst 的 prompt 模板注入了完整的上下文链（项目背景 + 调研结论 + 复杂度评估），这是跨模块分析的基础。
- 但 prompt 中没有显式的**跨模块依赖分析指令** -- 例如"识别前后端接口契约"、"列出数据库 schema 变更"、"评估认证流程的端到端延迟"。

**Step 1.5.5 决策协议**:
- 对 `large` 复杂度，强制使用完整决策卡片格式。预期产出的决策点：
  1. JWT vs Session-based 认证策略
  2. OAuth2.0 库选择（Passport.js vs NextAuth vs 自建）
  3. 密码存储方案（bcrypt vs argon2）
  4. RBAC 粒度（角色级 vs 权限级 vs ABAC）
  5. Token 刷新策略（滑动窗口 vs 固定过期）
- 决策卡片中要求"引用 research-findings.md 中的事实数据"确保推荐方案有数据支撑。

**Step 1.6 多轮决策 LOOP**:
- `large` 复杂度强制 3+ 轮 QA，强制苏格拉底模式。
- 苏格拉底模式的"强制排优"步骤（"如果只能实现 3 个，选哪些？"）对此场景特别有价值 -- 迫使用户在 MVP 阶段取舍（例如先做邮箱登录+JWT，延后 OAuth 和 RBAC）。
- "魔鬼代言人"步骤可能挑战"是否真的需要 RBAC，还是简单的角色区分就够了"。
- 循环内对 scope creep 的检测（功能点持续增加时主动提出 MVP 收敛）是一个重要机制。

**Step 1.7-1.8 结构化提示词与最终确认**:
- 提示词结构（背景目标 / 功能清单 / 决策结论 / 技术约束 / 技术方案 / 验收标准 / 影响范围）对跨模块需求是完备的。
- "验收标准"要求"可测试的验收条件"-- 对认证系统意味着需要明确如 "登录 API 响应 < 200ms"、"JWT token 过期后返回 401"等具体条件。

### 4.3 评分与发现

**得分: 8.0 / 10**

| 维度 | 发现 | 评价 |
|------|------|------|
| 并行调研架构 | 三路并行（Auto-Scan + 技术调研 + 联网搜索）对跨模块需求效率高 | 优势 |
| 安全需求识别 | 搜索决策规则引擎自动检测 "auth/JWT/安全" 关键词，强制搜索 | 优势 |
| 依赖评估深度 | 7 步调研的"依赖深度分析"覆盖 CVE/维护状态/许可证 | 优势 |
| 跨模块契约分析 | 缺少显式的前后端接口契约推导、数据库 schema 影响分析指令 | 缺陷 |
| 决策协议质量 | 强制完整卡片 + 事实数据引用 + scope creep 检测 | 优势 |
| 集成风险识别 | 风险识别在调研 Agent 中有覆盖，但没有专门的"集成测试策略前置讨论" | 风险 |

**关键缺陷 B-1**: 跨模块需求缺少显式的接口契约推导步骤 -- 认证系统涉及前端登录流程、后端 API 端点、Token 存储策略的多层协调，但 business-analyst 的 prompt 中没有强制要求产出接口定义或序列图。

**关键缺陷 B-2**: 安全需求的特殊处理不足 -- 虽然搜索决策规则引擎能识别安全关键词，但 Phase 1 没有安全威胁建模（threat modeling）的专项步骤。对认证系统这类安全敏感需求，应在需求阶段就考虑 OWASP Top 10 相关风险。

## 5. 场景 C 评审: 存量代码改造需求

### 5.1 输入设计

**需求输入**: "将现有的 REST API 迁移到 GraphQL，保持向后兼容，分阶段迁移（先查询后突变），前端逐步切换"

这是典型的存量代码改造需求：涉及大量现有代码的理解、兼容性保证、迁移策略设计、渐进式切换方案。

**评估重点**: Phase 1 的 Auto-Scan 和调研机制能否充分理解现有 REST API 的规模、复杂度和依赖关系，制定合理的迁移策略。

### 5.2 Phase 1 流程分析

**Step 1.2 Auto-Scan 存量代码调研**:

扫描范围表的覆盖度分析：

| 扫描项 | 对 REST→GraphQL 迁移的价值 | 覆盖度评估 |
|--------|---------------------------|-----------|
| 技术栈 | 识别后端框架（Spring Boot / Express / Fastify 等）→ 决定 GraphQL 集成方案 | 充分 |
| 目录结构 | 识别 API 路由文件、控制器、服务层的组织方式 | 充分 |
| 现有模式 | Grep 搜索路由定义、API 端点 → 建立 REST 端点完整清单 | 部分覆盖 |
| 依赖清单 | 识别现有 HTTP 框架依赖 → 确定 GraphQL 兼容性 | 充分 |
| 项目约束 | 从 CLAUDE.md 提取编码规范 | 充分 |
| 测试基础设施 | 识别现有 API 测试 → 迁移后需要保留/改造的测试 | 部分覆盖 |

**核心盲区**: Auto-Scan 的 Grep 模式搜索（路由定义、API 端点）能发现端点列表，但无法：
1. 解析 REST API 的请求/响应 schema -- GraphQL 迁移需要知道每个端点的输入输出结构
2. 识别 API 间的调用关系 -- 某些端点可能被其他端点内部调用（聚合 API）
3. 统计 API 的调用频率 -- 高频 API 应优先迁移
4. 识别 WebSocket/SSE 等非 REST 通信 -- 这些需要 GraphQL Subscription 方案

**Step 1.2 持久化上下文检查**: 对存量改造项目，如果之前已运行过 autopilot，持久化的 `project-context.md` 可能已过期（7 天有效期）。更重要的是，之前的上下文可能没有覆盖 REST API 的详细信息。增量扫描策略（"Grep 需求关键词相关文件"）对 "REST API 迁移" 的关键词可能不够精确。

**Step 1.3 技术调研**:
- Research Agent 的"相关代码影响分析"对此场景至关重要 -- 需要枚举所有 REST 端点及其处理函数。
- "依赖兼容性检查"将评估 GraphQL 库（Apollo Server / graphql-yoga / Mercurius）与现有框架的兼容性。
- "技术可行性评估"应覆盖：向后兼容方案（REST + GraphQL 共存）、schema 设计策略、N+1 查询问题。

联网搜索触发分析：
- `architecture_decision: true` -- GraphQL 迁移是架构决策，强制搜索
- `new_dependency: true` -- 引入 GraphQL 库
- 搜索查询如 "REST to GraphQL migration best practices 2026"、"GraphQL + {框架名} gradual migration" 价值明确

**Step 1.4 复杂度评估**:
- `total_files` 取决于 REST API 的规模，通常 > 5 个文件，路由到 `large`。
- 额外调整因子中的"3+ 新依赖"可能触发（graphql + apollo-server + graphql-tools 等）。

**Step 1.5 需求分析**:
- business-analyst 的 prompt 注入了 Steering Documents + research-findings.md，理论上可以基于调研结论制定迁移策略。
- 但 prompt 中没有针对**迁移类需求**的特殊指令 -- 例如"产出分阶段迁移计划"、"评估回滚策略"、"定义兼容性验收标准"。

**Step 1.6 多轮决策 LOOP**:
- 对迁移需求，预期决策点应包括：
  1. GraphQL 库选择（Apollo vs graphql-yoga vs Mercurius）
  2. Schema 设计策略（schema-first vs code-first）
  3. 向后兼容方案（REST 代理层 vs 并行运行 vs BFF 模式）
  4. 迁移顺序（按模块 vs 按端点类型 vs 按调用频率）
  5. 前端切换策略（全量切换 vs A/B 测试 vs Feature Flag）
- 苏格拉底模式的"探索替代方案"可能挑战"是否真的需要 GraphQL，还是 REST 优化就够了" -- 这是高价值提问。

**Step 1.7 结构化提示词**:
- "影响范围"章节要求预估影响文件 + 代码行数，这对迁移需求是合理的。
- 但缺少"迁移里程碑"和"回退计划"等迁移特有的结构化内容。

### 5.3 评分与发现

**得分: 6.5 / 10**

| 维度 | 发现 | 评价 |
|------|------|------|
| 存量代码扫描 | Glob + Grep 能发现端点列表，但无法解析 API schema 和调用关系 | 缺陷 |
| 迁移特化处理 | 流程中没有针对"迁移类需求"的特殊路径或模板 | 缺陷 |
| 调研深度 | `deep` 调研执行全部 7 项任务，依赖评估和同类实现对比价值高 | 优势 |
| 兼容性分析 | 没有显式的"向后兼容验证策略"前置讨论 | 缺陷 |
| 搜索策略 | `architecture_decision` 强制搜索，关键词构造规则合理 | 优势 |
| 分阶段规划 | business-analyst prompt 缺少迁移类需求的分阶段规划指令 | 缺陷 |

**关键缺陷 C-1**: Auto-Scan 对存量 API 的理解深度不足 -- Grep 搜索路由定义只能获取端点 URL，无法获取请求/响应结构、认证要求、分页策略等迁移关键信息。需要更深层的 AST 级或 OpenAPI spec 级分析。

**关键缺陷 C-2**: 缺少迁移类需求的特化流程 -- 当需求包含"迁移/migrate/替换/升级"等关键词时，应自动注入迁移特化模板（包含回退计划、兼容性矩阵、分阶段里程碑等结构化产出）。

**关键缺陷 C-3**: 历史知识注入（`.autopilot-knowledge.json`）对存量改造的支持有限 -- 知识格式为 pitfall/decision/pattern 标签，但没有"API 端点清单"或"架构依赖图"这类存量代码结构知识。

## 7. 与理想工作流的 Gap 分析

### 7.1 理想工作流定义

一个理想的 Phase 1 需求分析工作流应具备以下能力：

| 能力 | 理想状态 | Phase 1 现状 | Gap |
|------|---------|-------------|-----|
| 需求分类 | 自动识别需求类型（新功能/改造/修复/性能优化），按类型走差异化流程 | 不区分需求类型，统一走 10 步流程 | 中等 |
| 模糊度检测 | 需求进入时评估信息完备度，不足时强制澄清再调研 | 直接调研，无前置信息量门槛 | 显著 |
| 代码理解深度 | AST 级代码分析 + 调用图 + 数据流分析 | Glob + Grep + Read 的文本级扫描 | 显著 |
| 安全威胁建模 | 安全敏感需求自动触发 STRIDE/OWASP 威胁分析 | 仅在搜索策略中识别安全关键词，无专项威胁建模 | 中等 |
| 迁移规划 | 迁移类需求自动产出兼容性矩阵 + 回退计划 + 分阶段里程碑 | 通用流程，无迁移特化 | 中等 |
| 非功能需求 | 系统性覆盖性能/安全/可用性/可维护性/可观测性 | 在调研中部分覆盖（性能影响、安全隐患），但无结构化非功能需求清单 | 轻微 |
| 对话收敛保障 | 硬性轮数上限 + 信息熵收敛度检测 | 依赖 scope creep 检测 + 用户主动确认 | 轻微 |
| 跨模块接口契约 | 自动推导模块间接口定义（API contract / Event schema） | business-analyst 可能产出，但无强制要求 | 中等 |

### 7.2 Gap 优先级排序

1. **P0 -- 模糊度检测门槛**: 直接影响全流程效率，模糊需求下的调研浪费大量 token
2. **P1 -- 需求类型分类与差异化流程**: 改造类/优化类需求与新功能需求的分析侧重点完全不同
3. **P1 -- 存量代码深度分析**: Auto-Scan 的扫描精度限制了需求理解的上限
4. **P2 -- 跨模块接口契约推导**: 复杂需求的集成风险需要在需求阶段就识别
5. **P2 -- 安全威胁建模前置**: 安全敏感需求的专项分析不应延迟到实施阶段
6. **P3 -- 复杂度路由多维化**: 单一的 total_files 度量不足以准确反映复杂度

## 9. 改进建议

### 9.1 P0: 前置需求模糊度检测（推荐优先实施）

在 Step 1.1 获取需求来源之后、Step 1.2 Auto-Scan 之前，增加一个**确定性模糊度检测**步骤：

```
Step 1.1.5: 需求信息量评估（主线程执行）

规则引擎检测以下维度:
- 需求文本长度 < 20 字符 → flag: brevity
- 不包含任何技术名词（组件/API/模块/库名）→ flag: no_tech_entity
- 不包含量化指标（数字/百分比/时间单位）→ flag: no_metric
- 不包含具体动词（创建/迁移/修复/集成）→ flag: vague_action

IF flags.count >= 3:
    → 强制进入"需求澄清预循环"（在调研之前）
    → AskUserQuestion: 构造 3-5 个针对性澄清问题
    → 澄清完成后更新 RAW_REQUIREMENT，再进入调研
ELIF flags.count >= 2:
    → 标记 requirement_clarity: "low"
    → 调研 Agent prompt 追加 "需求模糊，请优先进行范围界定而非方案探索"
ELSE:
    → 正常流程
```

**预期收益**: 避免在 "优化性能" 这类模糊需求上浪费三路并行调研的 token，通过前置澄清使后续调研更有针对性。

### 9.2 P1: 需求类型分类与差异化路径

在模糊度检测之后增加需求类型分类：

```
Step 1.1.6: 需求类型分类（主线程执行）

通过关键词规则引擎分类:
- 含 "迁移/migrate/替换/replace/升级/upgrade" → type: migration
- 含 "优化/性能/performance/速度/latency" → type: optimization
- 含 "修复/fix/bug/defect" → type: bugfix
- 含 "新增/添加/创建/实现/add/implement" → type: new_feature
- 含 "重构/refactor/清理/cleanup" → type: refactoring
- 默认 → type: new_feature

基于类型注入差异化指令:
- migration: 追加兼容性矩阵模板 + 回退计划模板 + 分阶段里程碑
- optimization: 追加性能基线采集指令 + 量化目标确认
- bugfix: 简化流程，跳过联网搜索，聚焦复现步骤
- new_feature: 当前标准流程
- refactoring: 简化流程，聚焦代码影响范围分析
```

### 9.3 P1: 增强存量代码分析能力

在 Auto-Scan 扫描范围中增加以下扫描项：

| 扫描项 | 方法 | 产出 |
|--------|------|------|
| API 端点详情 | Grep: `@GetMapping/@PostMapping/@RequestMapping` (Java) 或 `router.get/post/put/delete` (Node) + Read 对应函数签名 | API 清单（URL + 方法 + 参数 + 返回类型） |
| OpenAPI/Swagger Spec | Glob: `**/swagger*.{json,yaml}`, `**/openapi*.{json,yaml}` + Read | 完整 API 契约 |
| 数据库 Schema | Glob: `**/migrations/**`, `**/schema.prisma`, `**/entities/*.{ts,java}` + Read | 数据模型清单 |
| 模块间导入关系 | Grep: `import.*from` / `require(` / `@Autowired` | 模块依赖热图（哪些模块被最多引用） |

### 9.4 P2: 复杂度路由多维化

将复杂度评估从单一 `total_files` 扩展为加权多维评分：

```
complexity_score = (
    total_files * 1.0
  + new_dependencies * 2.0
  + cross_module_boundaries * 3.0    # 跨越的模块边界数
  + schema_changes * 2.5              # 数据库 schema 变更数
  + api_surface_changes * 2.0         # API 端点新增/修改数
  + security_sensitivity * 3.0        # 涉及认证/加密/权限
)

IF complexity_score <= 4:  → small
ELIF complexity_score <= 12: → medium
ELSE: → large
```

### 9.5 P2: 安全敏感需求前置威胁建模

当搜索决策规则引擎识别到 `security_related: true` 时，在 business-analyst dispatch 中追加安全分析指令：

```
## 安全威胁建模（当需求涉及认证/授权/加密/数据保护时）

对每个安全相关功能点，执行 STRIDE 分析:
- Spoofing（身份伪造）: 攻击者能否冒充合法用户？
- Tampering（篡改）: 数据在传输/存储中能否被篡改？
- Repudiation（抵赖）: 是否有审计日志？
- Information Disclosure（信息泄露）: 敏感数据是否可能泄露？
- Denial of Service（拒绝服务）: 是否有速率限制？
- Elevation of Privilege（权限提升）: 用户能否越权操作？

产出: 威胁清单 + 缓解策略 → 作为决策点呈现给用户
```

### 9.6 P3: 对话收敛硬上限

在多轮决策循环中增加硬性上限：

```
max_rounds = config.phases.requirements.max_decision_rounds (默认 8)

IF current_round >= max_rounds:
    → AskUserQuestion: "已进行 {max_rounds} 轮讨论，仍有 {N} 个未决策点。
       建议: 将未决策点标记为 'deferred'，在实施阶段遇到时再确认。
       选项: [接受建议，继续] / [继续讨论（追加 3 轮）]"
```
