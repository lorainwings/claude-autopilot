## Critical 问题（4 个，需修复）

### C1: `configuration.md` 缺失两个顶级字段文档

**来源**: Full 模式 Agent
**位置**: `docs/configuration.md`

`default_mode` 和 `background_agent_timeout_minutes` 在 `config-schema.md` 和 `SKILL.md` 中被引用，但 `configuration.md` 完全未记录。同时 `config-schema.md` 的 YAML 模板中也缺少 `default_mode` 字段。

**修复**:
1. `docs/configuration.md` Top-Level Fields 表新增 `default_mode` (string, optional, default: "full") 和 `background_agent_timeout_minutes` (number, optional, default: 30)
2. `config-schema.md` 模板中 `version` 之后添加 `default_mode: "full"`

### C2: `validate-config.sh` 缺失两个字段的验证规则

**来源**: Full 模式 Agent
**位置**: `scripts/validate-config.sh` TYPE_RULES + RANGE_RULES

`default_mode` 和 `background_agent_timeout_minutes` 在运行时被使用但未在 validate-config.sh 中验证。

**修复**:
- TYPE_RULES 新增 `'default_mode': str`, `'background_agent_timeout_minutes': (int, float)`
- RANGE_RULES 新增 `'background_agent_timeout_minutes': (1, 120)`
- 可选: default_mode 枚举值检查 (full/lite/minimal)

### C3: Phase 6 后台 dispatch 绕过 L2 门禁

**来源**: Lite 模式 Agent（cross-confirmed by 整体流程 Agent）
**位置**: `scripts/check-predecessor-checkpoint.sh:52-54`

Phase 6 路径 A 使用 `run_in_background: true` 派发，触发 Layer 1.5 bypass (`exit 0`)，导致 Phase 6 的 **全部** L2 前置校验被跳过（包括 `zero_skip_check` 和 `tasks completion check`）。在 lite 模式下，确定性门禁（L2）被架空，完全依赖 AI 侧 L3。

**修复方案（二选一）**:
- **方案 A**: Layer 1.5 bypass 中仅跳过 predecessor checkpoint 检查，保留 Phase 6 特有的 zero_skip_check + tasks completion check
- **方案 B**: 接受现状，在 `autopilot-gate/SKILL.md` 中将 Phase 5→6 的 tasks completion + zero_skip_check 标记为 L3 **硬阻断**（不可跳过），并在 `check-predecessor-checkpoint.sh` 注释中说明设计意图

### C4: `write-edit-constraint-check.sh` TDD 模式 Phase 5 识别脆弱性

**来源**: 整体流程 Agent + Full 模式 Agent
**位置**: `scripts/write-edit-constraint-check.sh:39-67`

Phase 5 检测逻辑依赖 Phase 4 checkpoint 存在性判断执行模式。TDD 模式下 Phase 4 写入 `phase-4-tdd-override.json` 后可正确识别，但若该 checkpoint 写入失败或延迟，脚本会进入 lite/minimal 分支逻辑，导致 Phase 5 Write/Edit 约束检查可能误判。

**修复**: 增加 TDD 模式显式识别 — 读取锁文件 mode 字段 + `read_config_value` tdd_mode，不完全依赖 Phase 4 checkpoint 存在性。

## Warning 问题（6 个，建议修复）

### W1: `get_predecessor_phase()` 冗余 tdd_mode 读取

**来源**: 三个模式 Agent 均报告
**位置**: `scripts/check-predecessor-checkpoint.sh:207-209`

`read_config_value("phases.implementation.tdd_mode")` 在函数入口无条件调用，但 lite/minimal 分支不使用该值。每次调用产生 ~50-100ms 的 python3 fork 开销。另外在 Phase 5 特殊门禁段（~line 278）再次读取，造成二次冗余。

**修复**: 将 tdd_mode 读取移到 `full` 模式分支内部，或在主逻辑入口读取一次赋值为全局变量复用。

### W2: `configuration.md` 文档不一致

**来源**: Full 模式 Agent + 整体流程 Agent
**位置**: `docs/configuration.md`

两处不一致：
1. `parallel.max_agents` 默认值文档写 `3`，但 `config-schema.md` 和 `SKILL.md` 均为 `8`
2. 范围验证规则表缺少 6+ 个 v3.6.0 新增字段（wall_clock_timeout_hours, hook_floors.*, min_total_cases）
3. 类型验证规则表缺少 12+ 个在 validate-config.sh 中实际存在的条目

**修复**: 统一 max_agents 默认值为 8，补充完整的类型/范围验证规则到文档。

### W3: 并行 TDD 缺少 L2 确定性验证

**来源**: 整体流程 Agent
**位置**: `references/tdd-cycle.md` 并行 TDD 章节

并行 TDD 模式下，域 Agent 的 RED-GREEN-REFACTOR 完全由 AI 自主执行（prompt 注入），缺少 L2 确定性验证。Agent 可能虚报 `tdd_cycles` 中的 `red_verified`/`green_verified`。

**修复**: 在并行 TDD 域 Agent 合并后，主线程增加 L2 验证步骤 — 运行 `full_test_command` 验证所有测试确实通过，对比 tdd_cycles 报告与实际测试结果。

### W4: `autopilot-gate/SKILL.md` Phase 1→5 门禁描述不准确

**来源**: Minimal 模式 Agent
**位置**: `skills/autopilot-gate/SKILL.md:189`

描述为 "Phase 1 checkpoint status 为 ok"，但 `check-predecessor-checkpoint.sh` 实际同时接受 ok 和 warning。

**修复**: 修改为 "status 为 ok 或 warning"。

### W5: `validate-decision-format.sh` 实际触发范围为零

**来源**: Minimal 模式 Agent
**位置**: `scripts/validate-decision-format.sh` + `skills/autopilot-dispatch/SKILL.md`

该 Hook 仅检查含 `<!-- autopilot-phase:1 -->` 标记的 Task，但根据 dispatch 协议，Phase 1 的 research/business-analyst Task **均不含 autopilot-phase 标记**。因此该 Hook 在当前设计下从不触发。

**修复**: 在文档中注明这是设计意图（Phase 1 走主线程交互模式），或考虑调整 Phase 1 最终信封的验证方式。

### W6: bash/Python 版 `read_config_value` 对非标量值行为差异

**来源**: 整体流程 Agent
**位置**: `scripts/_common.sh:154-231` vs `scripts/_envelope_parser.py:130-190`

bash 版返回 Python `str()` 表示（如字典/列表变为字符串），Python 版返回原始对象。对标量值无影响，但对嵌套值行为不同。

**修复**: 为 bash 版添加文档注释"仅用于读取标量值"，并考虑统一 `validate-json-envelope.sh` 中的 `read_hook_floor()` 为 importlib 导入版本（消除第三个实现）。
