# Role: Full-Stack Architect & DX (Developer Experience) Lead

# Objective: 为 `spec-autopilot` 插件搭建面向 GUI 的通信桥梁，并实现一套“开箱即用”的 GUI 分发与服务机制。确保终端用户无需手动配置前端项目，即可在浏览器中自动打开可视化大盘

## ⚠️ 核心执行纪律 (Rules of Engagement)

1. **零侵入原则:** 引擎侧代码、GUI 静态资产必须与用户当前的业务项目绝对隔离。日志仅写入业务目录下的 `.autopilot/`。
2. **内聚分发 (Embedded GUI):** 必须设计一种机制，使前端 React 代码编译后的产物能够随插件一同分发，并由插件自带的服务器进行托管。
3. **DX 优先:** 插件启动时，必须做到静默启动后端服务、自动抛出本地 URL、并尽可能自动打开浏览器。

---

## 🛠️ 桥接与体验优化任务流 (Bridge & DX Pipeline)

### [Task 1] 引擎侧：事件 Payload 增强 (Event Schema Upgrade)

- **动作:** 修改 `emit-phase-event.sh` 和 `emit-gate-event.sh`。
- **要求:** 确保 JSON Payload 中强制注入 `change_name`, `session_id`, `phase_label`, `total_phases`, 以及自增的 `sequence`，为 UI 渲染提供充足上下文。

### [Task 2] 中间件：双模服务器搭建 (Dual-Mode Server)

- **动作:** 在引擎侧创建 `scripts/autopilot-server.ts` (替代之前的简单 websocket server)。
- **职责:** 1. **WebSocket 服务 (端口 8765):** 负责拦截子进程 CLI 终端流，以及监听 `.autopilot/events.jsonl`，并向前端推送。
  2. **HTTP 静态服务 (端口 9527):** 拦截指定的本地目录（如插件根目录下的 `gui-dist/`），将其作为静态 Web 网站托管。
  3. **生命周期钩子:** 服务启动成功后，使用 Node.js 的 `open` 或子进程调用，尝试自动在默认浏览器中打开 `http://localhost:9527`。

### [Task 3] 前端工程：独立开发与打包流 (Frontend Scaffolding & Build)

- **动作:** 在本地独立创建 `vibe-workflow-app/` 目录初始化 React/Zustand 项目。
- **职责:**
  1. 编写 `WSBridge` 类连接 `ws://localhost:8765`。
  2. 实现 `PhaseTimeline`, `ParallelKanban`, `GateBlockCard` 以及引入 `xterm.js` 渲染终端流的 `VirtualTerminal`。
  3. **【关键配置】:** 配置打包工具（如 Vite/Webpack），使其 `build` 命令的输出目录直接指向 `spec-autopilot` 插件仓库的 `gui-dist/` 目录，实现一次编译，两端同步。

### [Task 4] 启动口令整合 (Command Integration)

- **动作:** 将服务器启动逻辑无缝融合到插件的现有启动指令中。
- **职责:** 当用户运行插件的入口脚本（如 Phase 0 启动时），后台自动检测 `autopilot-server.ts` 是否存活，若未存活则作为守护进程 (Daemon) 启动。确保用户在终端看到的只有一行优雅的提示：“✨ 引擎已启动，GUI 大盘见 <http://localhost:9527”。>
