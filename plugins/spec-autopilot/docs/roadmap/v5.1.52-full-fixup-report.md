  按你 8 个维度的结论

  1. full/lite/minimal 全流程仿真
     lite/minimal 基础测试能过，但模式安全性并不完整，因为启动扫描、压缩状态、恢复建议没有统一使用 mode-aware phase graph。full 也
     缺少与 lite/minimal 对等的端到端混沌恢复回归。优先修复统一 phase graph，再补 full/lite/minimal 三套黑盒回放用例。
  2. 主窗口必要信息与 AI 编排最佳实践
     主窗口必要信息应该只有控制面数据：当前 change/session/mode、phase/sub-step、阻塞 gate、待确认决策、agent 责任边界、下一不可逆
     动作、上下文预算、恢复来源。cwd/worktree/transcript/cost 不是首要信息，属于诊断补充。最佳实践是把主窗口做成“单一编排控制台”，
     而不是事件瀑布流。
  3. 上下文压缩是否完全受控
     结论是否定的。当前是摘要回灌，不是完整恢复，也没有 restore verification，因此不能证明“回复后召回内容符合预期”。修复要点是
     manifest、hash、restore recipe、恢复后自检。
  4. 崩溃恢复是否符合预期
     不完全符合。恢复逻辑分裂、gap 语义不一致、fallback 依赖 mtime、清理依赖 substring，都说明 crash recovery 还不是严格受控事务。
     建议把恢复设计成 journal-driven state machine，并明确 checkpoint committed / artifact committed / gate committed 三类原子边
     界。
  5. 需求评审与三路 agent 调研是否必要
     三路调研不应默认必要，应按风险自适应触发。更重要的是留下稳定的证据链，而不是增加 agent 数量。要让 AI 更透彻理解需求，关键不
     是“多路”，而是强制沉淀 澄清问题、冲突点、来源、决策理由、未决假设 并冻结到后续 phases 可引用的结构化工件里。
  6. OpenSpec 与 OpenSpec FF 是否有异常风险
     有，而且核心风险不是流程本身，而是 validator 太弱。只要 Phase 2/3 的 envelope 还能空心通过，FF 就可能把问题带到实现阶段。修复
     重点是 phase-specific hard gates，而不是再补更多文档说明。
  7. 测试用例是否贴合产品需求、如何避免 Hack Reward
     目前部分测试更贴近文档和兼容壳层，不够贴近真实产品流。合理性和覆盖性要靠“场景矩阵 + 失败注入 + 黑盒验收”，独立性要靠每次测试
     独立 temp workspace、独立 change_id/session_id、禁止共享历史状态，避免 Hack Reward 要靠只认真实入口和真实副作用，不认 grep 文
     本。
  8. 是否严格遵守 rules 与全局 CLAUDE.md
     现在无法严格证明。系统最多证明“没有明显违反某些禁令”，但不能证明“真的使用了项目要求的架构与规范”。要补的是 provenance-based
     compliance，而不是继续堆 prompt 文字。

  建议 Claude 的修复落地顺序

  1. 先修控制面一致性
     统一 phase 推断逻辑，把 scan-checkpoints、save-state-before-compact、recovery-decision 收敛到同一个 mode-aware phase graph
     库。
  2. 再修 runtime hard gates
     把 Phase 1、Phase 2、Phase 3 的 validator 升级为强校验，补 decisions/artifacts/provenance 必填；取消 marker 才校验的旁路设
     计。
  3. 再修 event bus
     把 sequence、event_id、request_id、gate_id、session_id 全部补齐，并改成原子写入；修完后先让并发事件测试稳定通过。
  4. 再修 GUI 控制台
     主窗口改为 orchestration-first，决策链路加入 request correlation 和 ACK 状态机，避免多窗口/重复 block 串扰。
  5. 再修 compaction / crash recovery
     引入 manifest、hash、restore verification、stable ID 绑定，去掉 mtime fallback 和 substring cleanup。
  6. 最后重构测试体系与合规证明
     把测试从 doc/compat shell 拉回到 live hooks/black-box flow；同时补 inputs_read/rules_applied/spec_refs_used 的合规证明字段和
     校验。
