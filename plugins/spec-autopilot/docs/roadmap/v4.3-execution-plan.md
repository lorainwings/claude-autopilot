# Role: QA Lead & Vibe Workflow Integration Architect (QA 负责人 & 可视化工作流集成架构师)

# Objective: 对 `spec-autopilot` 新交付的 V4.2 补丁进行“破坏性验证 (Acid Test)”，解决全局版本号同步的遗漏问题，并在确认底层防御坚如磐石后，正式输出面向 `vibe-workflow` 图形化界面的集成白皮书

## ⚠️ 核心执行纪律 (Rules of Engagement)

1. **零信任原则 (Zero Trust):** 不要相信代码逻辑，只相信测试结果。必须构造极端的异常场景来触发新写的防御 Hook。
2. **机制取代记忆 (Mechanism over Memory):** 严禁依赖 AI 的上下文记忆来同步状态（如版本号），必须通过确定性的脚本来强制保障一致性。
3. **严格串行:** 必须按照 Task 1 到 Task 5 的顺序执行，前置验证失败必须停下报告，不可带病推进。

---

## 🛠️ 破坏性验证与架构跨界任务流 (Acid Test & Integration Pipeline)

### [Task 1] 规约防御的破坏性实测 (Defensive Hooks Acid Test)

- **目标:** 验证 V4.2 新增的防偷懒机制是否具备真正的绝对阻断力。
- **动作:**
  1. **TODO 逃逸攻击:** 强制生成一段包含 `// TODO: implement later` 或 `FIXME` 的代码，触发 `banned-patterns-check.sh`。预期结果：进程必须 `exit 1` 并打回重做。
  2. **恒真断言攻击:** 在测试文件中强行写入 `expect(1).toBe(1)` 或无意义的 Mock 验证，触发 `assertion-quality-check.sh`。预期结果：识别并硬阻断。
  3. **Sad Path 缺失攻击:** 提交一个只有 `happy_path_count: 5` 而 `sad_path_count: 0` 的 Phase 4 JSON，验证 Schema 校验是否报错。

### [Task 2] Phase 5 并发引擎压测 (Concurrency Stress Test)

- **目标:** 验证并行派发逻辑（P0-PERF-1 的修复）的稳定性和加速比。
- **动作:** 构造包含至少 5 个独立子任务的 Phase 5 需求。监控合并时的状态机是否混乱，文件是否发生竞态冲突，并量化时间优化比例。

### [Task 3] 全局版本号确定性同步机制 (Deterministic Version Sync)

- **目标:** 彻底杜绝发版时 `plugin.json`、`CHANGELOG.md`、`marketplace.json` 和 `README.md` 版本号不一致的技术债务。
- **动作:**
  1. **编写发版脚本:** 在 `scripts/` 目录下创建一个确定性的同步脚本（如 `bump-version.sh <new_version>`）。
  2. **硬编码解析:** 脚本必须利用 `jq` 和 `sed` 等工具，实现一键同步修改：
     - `plugin.json` 中的 `version` 字段。
     - 跨级寻址修改 `marketplace.json` 中 `plugins` 数组下对应本插件的 `version` 字段。
     - 正则替换 `README.md` 中的 Version Badge（例如 `![version](https://img.shields.io/badge/version-vX.X.X-blue)`）。
     - 在 `CHANGELOG.md` 顶部自动插入新版本的 Header。
  3. **强制收口:** 修改 `CLAUDE.md`，规定以后所有的版本升级必须且只能通过执行此脚本完成，禁止人工或 AI 散弹式手动修改。

### [Task 4] 事件总线“抓包”与契约校验 (Event Bus Sniffing)

- **目标:** 验收 V5.0 引擎抛出的 GUI 事件流是否可用于生产环境。
- **动作:** 收集前序任务运行期间抛出的 JSON 事件数据。以 React/Vue 前端开发者的挑剔视角，审查 Payload 是否包含渲染进度条、状态灯、报错弹窗所需的全部准确字段（`timestamp`, `phase_id`, `status`, `error_details` 等），并指出缺失项。

### [Task 5] `vibe-workflow` 集成白皮书与架构蓝图 (Integration Blueprint)

- **目标:** 基于验证通过的底层事件流，正式规划上层 GUI 工具的架构。
- **动作:**
  1. **架构分层:** 设计 `spec-autopilot` (底层引擎) 与 `vibe-workflow` (GUI 界面) 的通信桥梁（如本地 WebSocket 监听层）。
  2. **交互流转:** 描述底层触发 `GateBlock` 事件时，GUI 层应如何弹出交互式决策卡片并将指令注回底层。
  3. **Figma UI 映射:** 针对 Phase 进度、并发任务看板，提供直观的 UI 组件化设计建议。
- **输出:** 生成 `docs/reports/vibe-workflow-integration-blueprint.md`。
