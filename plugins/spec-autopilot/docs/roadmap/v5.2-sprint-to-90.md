## 🛠️ v5.2 卓越冲刺重构任务流 (The Path to 90+)

### [Task 1] 性能与 Token 极速瘦身 (Performance Optimization)

- **背景:** `parallel-dispatch.md` (33KB) 在 4 个阶段被全量注入，造成了 13-17% 的无谓 Token 浪费。`post-task-validator` 默认 150s 超时过长。
- **动作:**
  1. 将 `references/parallel-dispatch.md` 按阶段拆分为 4 个独立的小文件（如 `parallel-phase1.md`, `parallel-phase4.md` 等），并修改相关 `SKILL.md` 的注入引用，实现**按需加载**。
  2. 修改 `hooks.json`，将 `post-task-validator.sh` 的 timeout 从 `150` 降至 `60`，缩短最坏情况下的阻塞时间。

### [Task 2] GUI 实时性与双向闭环终极加固 (GUI & Telemetry)

- **背景:** Phase 5 细粒度进度缺失；终端无颜色；决策卡片点击后无确认感。
- **动作:**
  1. **进度事件:** 在引擎侧实现 `TaskProgressEvent` 的发射（在 Phase 5 每个 task 完成后），并在 GUI 前端的 `ParallelKanban.tsx` 和 Store 中增加对该事件的精确消费，让并发看板真正跳动起来。
  2. **决策 ACK:** 在 `autopilot-server.ts` 收到 GUI 的决策并写入文件后，立刻通过 WebSocket 广播一个 `decision_ack` 事件，前端收到后立即销毁/隐藏 GateBlockCard。
  3. **终端着色:** 审查并修复 `VirtualTerminal.tsx`，确保传入后端的 ANSI 转义码能被 `xterm.js` 正确解析和上色，或在前端对特定事件（如 block/error）手动赋予颜色标签。

### [Task 3] 规约与 TDD 确定性补漏 (Compliance & TDD Ironclad)

- **背景:** TDD 阶段的 Refactor 失败回滚（TDD-5）仅靠 AI 自觉，无 L2 兜底；崩溃恢复未清理 `.tdd-stage` 导致状态污染。
- **动作:**
  1. 在 `autopilot-recovery/SKILL.md` 的恢复清理步骤中，显式增加 `rm -f openspec/changes/*/context/.tdd-stage`，防止状态残留。
  2. 升级 TDD-5 强制力：在主线程串行 TDD 流转逻辑或 `_post_task_validator.py` 中，增加硬编码逻辑：如果 Refactor 阶段测试失败（`reverted: true`），必须强制执行 `git checkout -- .` 以确保代码绝对不被污染。
  3. 扩展 `_post_task_validator.py` 的反合理化正则，增加 3 个高频借口模式：时间/Deadline 借口、环境配置借口、第三方依赖阻塞借口。

### [Task 4] Phase 1 需求引擎进化 (Phase 1 Engineering)

- **背景:** 无法处理“重构+新功能”的复合需求；缺失分布式/高并发的非功能需求挖掘；`min_qa_rounds` 配置沦为死代码。
- **动作:**
  1. 修改 `phase1-requirements.md` 中的路由分类逻辑，允许复合需求（如 `requirement_type: ["feature", "refactor"]`），并合并获取最严苛的 `routing_overrides`（如 100% 测试覆盖率）。
  2. 在 `phase1-supplementary.md` 的苏格拉底模式中，增加第 7 步专项追问：“非功能需求质询 (SLA/性能/可靠性)”，当检测到并发/分布式等关键词时强制触发。
  3. 让 `min_qa_rounds` 配置生效：在 Phase 1 的 Step 1.6 多轮决策循环逻辑中，读取 `config.phases.requirements.min_qa_rounds` 作为强制最低 QA 轮数下限。
