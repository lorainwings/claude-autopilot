### 🔴 P0 级致命缺陷修复 (Fatal & Critical)

#### [Task 1] 修复双向反控路径断裂 (DC-PATH)

- **背景:** GUI 点击 Override 后，服务端将决策写入 `{projectRoot}/.autopilot/decision.json`，而引擎侧 `poll-gate-decision.sh` 却在 `{change_dir}/context/decision.json` 苦苦死等，导致反控完全失效。
- **动作:** 统一决策文件路径。请修改 `scripts/autopilot-server.ts` (约 41 行)。使其在处理 decision 写入时，能够通过读取项目根目录下的 `.autopilot-active` 锁文件推导出当前的 `change_dir`，并将决策正确写入到 `{change_dir}/context/decision.json` 中，与 `poll-gate-decision.sh` 保持严格一致。

#### [Task 2] 修复 IN_PHASE5 误判拦截 (D-01)

- **背景:** `unified-write-edit-check.sh` (约 42-71 行) 的 Phase 5 检测逻辑存在漏洞。在 full 模式的 Phase 2/3 期间，会错误落入第三分支，导致 Phase 2/3 子 Agent 写入 context 的操作被误拦截。
- **动作:** 优化该脚本的 `IN_PHASE5` 判定逻辑。在 `elif [ -n "$PHASE1_CP" ];` 分支中，增加对当前模式（从锁文件读取）及 Phase 3/4 checkpoint 存在性的严格校验，确保 full 模式下未到达 Phase 5 时绝对不会触发 CHECK 0 拦截。

#### [Task 3] 修复 Python3 缺失时的 Fail-Open 漏洞 (D-05)

- **背景:** `post-task-validator.sh` (约 28-29 行) 在 python3 缺失时执行 `exit 0`，导致致命的静默跳过（静默放行所有 5 个验证器）。
- **动作:** 将 `if ! require_python3; then exit 0; fi` 改为 Fail-Closed 模式。当检测不到 python3 时，必须输出一段包含 `{"decision":"block", "reason":"python3 required"}` 的 JSON 到 stdout，并以适当的退出码终止，确保 Claude Code 会阻断流程而非放行。
