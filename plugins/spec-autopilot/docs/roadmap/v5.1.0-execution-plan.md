# 角色与目标

你是一位顶级的 AI 研发架构师（Vibe Coder），当前任务是修复并优化 `spec-autopilot` 插件在 v5.3 版本全面审计中暴露的 P0 与 P1 级缺陷。我们的目标是通过精准的定点修复，将系统评估总分从 87.3 提升至 95 分以上，彻底夯实 L2 层的确定性与 GUI 的鲁棒性。

# 🛑 【绝对红线：测试规范与代码提交流程】

在处理本次优化以及提交代码的 Hook 环节中，必须将以下原则作为最高执行指令：

1. **禁止“妥协式”修改**：当遇到测试不通过时，**绝对不允许**为了让测试强行通过，而随意修改插件核心的编排逻辑（如 `SKILL.md` 等）和已有的稳定实现。
2. **测试必须有意义**：对于每一个新修复的特性或约束，必须按照新特性**针对性地新增或修复测试用例**。
3. **规范化测试流程**：测试用例的编写绝不能流于形式，必须真实反映业务的边界与约束条件。优先审查测试用例的断言逻辑是否准确，而非盲目改动业务代码。

## Step 1: 补齐 P0 级测试基建盲区

当前测试套件虽然通过率 100%，但存在严重的核心路径测试缺失。请在 `tests/` 目录下新增并完善以下测试模块，严格遵守上述【绝对红线】：

* **路由动态阈值测试 (`test_routing_overrides.sh`)**: 模拟 Phase 1 checkpoint 写入不同的 `routing_overrides`（如 Bugfix 场景下 `sad_path_min_ratio_pct: 40` 和 `change_coverage_min_pct: 100`），验证 Phase 4 的 L2 Hook 是否正确应用了这些动态阈值而非默认值。
* **TDD 核心路径测试 (`test_tdd_isolation.sh`)**: 覆盖 RED/GREEN/REFACTOR 三个阶段对测试/实现文件的写入拦截（`.tdd-stage` 隔离逻辑），以及 TDD Metrics 的 L2 检查。
* **读写约束专项测试 (`test_unified_write_edit.sh` & `test_code_constraint.sh`)**: 补充对 `unified-write-edit-check.sh` 中 TODO/FIXME/HACK 拦截、恒真断言拦截的端到端测试，以及对 `code-constraint-check.sh` 阻断行为的专项测试。

## Step 2: 强化 L2 确定性硬约束 (P0)

当前部分规则过于依赖 AI 自律，需要下沉到 L2 Hook 强制阻断：

* **`min_qa_rounds` 强制验证**: 在 `_config_validator.py` 的 `RANGE_RULES` 中增加 `(1, 10)` 的范围约束。同时在 Phase 1 的 `_post_task_validator.py` 中，通过解析 `decisions` 数组或相关元数据，实现对 `min_qa_rounds` 实际执行轮数的硬校验。
* **`required_test_types` L2 校验**: 在 `_post_task_validator.py` (Phase 4) 中，读取 `routing_overrides.required_test_types`，强制验证信封的 `test_counts` 是否真实包含了路由要求的测试类型。
* **TDD 回滚自动化**: 创建 `scripts/tdd-refactor-rollback.sh` 封装 `git checkout -- .` 回滚逻辑，使 TDD-5 的回滚动作从 L3 决策降级为确定的脚本调用。

## Step 3: GUI V2 鲁棒性与性能治理 (P0 & P1)

增强可视化大盘的安全感与渲染性能：

* **WebSocket 容错处理**: 在服务端 `broadcastNewEvents` 中的 `JSON.parse(line)` 必须包裹 `try-catch`，避免单行日志格式错误引发的广播大面积中断。
* **决策反馈机制 (GUI)**: 解决 decision 发送失败无 UI 反馈的遗留风险，在 `GateBlockCard` 中增加发送失败的 Toast 或内联红框提示。
* **服务端 I/O 优化**: 将 `getEventLines()` 改为基于 `lastLineCount` 偏移量的增量读取策略，解决高频写入时的 I/O 放大问题。
* **Zustand Selector 性能优化**: 为 `selectPhaseDurations`、`selectTotalElapsedMs` 等高频派生状态应用 memoization（如 `useMemo` 或 `useShallow`），消除 1 秒定时器引发的无谓重渲染开销。
