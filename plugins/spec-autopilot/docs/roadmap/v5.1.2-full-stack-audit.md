3. **全生命周期覆盖**：审计必须严格按照 Phase 0 到 Phase 7 的时间轴展开，不遗漏任何一个过场阶段和分支路径（Full / Lite / Minimal 模式）。

## 🛠️ 五维全栈审计矩阵 (5-Dimensional Audit Matrix)

请按照以下五个维度，逐一深入系统进行极限压测与源码审查：

### 维度一：编排与状态机流转 (Phase 0 -> Phase 7)

* **Phase 1 (需求与质询)**：验证 Socratic 7 步质询引擎是否完备。**重点核实：** `min_qa_rounds` 是否从“AI 软约束”真正升级为了 `_post_task_validator.py` 中的 L2 硬阻断？路由矩阵（Bugfix/Refactor 等）是否正确生成并下发？
* **Phase 4 & 5 (测试与实施)**：验证动态门禁阈值（`change_coverage`, `sad_path` 等）的传递链条是否完整。并行任务的 `parallel-phase*.md` 按需加载是否真实生效，Token 消耗是否符合预期？
* **全模式兼容性**：验证系统在 Full、Lite、Minimal 三种模式下的阶段跳跃是否稳健，前驱 Checkpoint 检查是否有漏洞？

### 维度二：L2 确定性铁壁 (The "Iron Guard" Determinism)

* **Write/Edit 拦截网**：深度审查 `unified-write-edit-check.sh` 和 `code-constraint-check.sh`。恒真断言、TODO 占位符、全局禁用词的拦截逻辑是否无懈可击？
* **反合理化引擎 (Anti-Rationalization)**：查阅 `_post_task_validator.py`，确认 29 种中英双语借口模式的正则表达式与计分阻断逻辑。
* **新增测试资产核实**：**重点检查**新引入的 `test_routing_overrides.sh`、`test_tdd_isolation.sh` 以及 `test_unified_write_edit.sh`。这些测试是真的在测试边界，还是流于形式？

### 维度三：TDD 引擎与代码生成 (TDD Engine & Code Generation)

* **生命周期隔离**：审视 `.tdd-stage` 文件的创建、消费与销毁链路。RED 阶段是否死锁了实现文件的写入？GREEN 阶段是否锁死了测试文件的篡改？
* **自动回滚机制**：核实 `scripts/tdd-refactor-rollback.sh` 是否已就位，彻底将 REFACTOR 失败时的 `git checkout -- .` 从 AI 的意图降维打击成了脚本级的绝对执行。
* **崩溃恢复**：验证 `autopilot-recovery` 技能在面对中断时，对 `.tdd-stage` 和 `.tmp` 残留的清理能力。

### 维度四：遥测、并发与底层基建 (Telemetry & Infrastructure)

* **原子性操作**：审查 `_common.sh` 中的 `next_event_sequence()`，确认 `flock` 排他锁的子 shell 实现是否足以应对 Phase 5 高并发（8 Agent）的写入冲击。
* **服务端 I/O**：审查 `autopilot-server.ts`。`getEventLines()` 是否已经实现了基于偏移量的增量读取？`broadcastNewEvents` 中的 `JSON.parse` 是否加固了 `try-catch`？
* **产物纯净度**：运行 `build-dist.sh`，核实最终产物是否绝对隔离了源码、测试文件和 `CLAUDE.md` 的开发专属段落。

### 维度五：GUI 交互与人机协同 (GUI & Human-in-the-Loop)

* **状态机竞态**：审查 `store/index.ts`。在高速事件流下，`decisionAcked` 的重置逻辑、`Set` 去重机制、以及 `.slice(-1000)` 的截断策略是否会导致界面状态闪烁或错乱？
* **容错与反馈**：审查 `GateBlockCard.tsx`。当用户点击决策且 WebSocket 发送失败时，系统是否提供了明确的 UI 报警？
* **渲染性能**：检查高频 selector（如 `selectPhaseDurations`）是否实现了 memoization。`VirtualTerminal` 的 ref 增量渲染是否依然高效？
