## 🛠️ 核心任务流 (Execution Pipeline)

### [Task 1] 规约终极补漏 (The "No Loopholes" Patch)

- **目标:** 彻底封死 AI 偷懒与伪造测试的路径 (对应报告 TD-1, TD-2, TD-3)。
- **动作:**
  1. **拦截 TODO/FIXME:** 在 L2 Hook 中新增 `banned-patterns-check.sh`。使用 `grep` 扫描代码变更，一旦发现 `TODO:`, `FIXME:`, `HACK:` 占位符，强制 `exit 1` 阻断。
  2. **拦截恒真断言:** 新增 `assertion-quality-check.sh`。使用正则或简单的 AST 分析，拦截类似 `expect(true).toBe(true)` 或 `assert True` 这种毫无意义的欺骗性断言。
  3. **量化 Sad Path:** 修改 Phase 4 (测试设计) 的 JSON Schema，将原本单一的 `test_counts` 强制拆分为 `happy_path_count` 和 `sad_path_count`，要求 AI 必须设计异常分支用例。

### [Task 2] Phase 5 串行性能破局 (Concurrency Engine)

- **目标:** 解决 P0-PERF-1，将耗时最长的 Phase 5 (代码生成) 从串行转为后台并行计算。
- **动作:**
  1. 升级 `parallel-dispatch.md`，实现真正的并行派发逻辑。
  2. 将 10 个 Task 拆分为可并发执行的子进程或并发 Agent Session，并在最后增加一个 `wait` 与 `merge` 的后置 Hook 来汇总并发结果。

### [Task 3] 需求分类路由 (Requirement Intelligence)

- **目标:** 落实 P1-REQ-2 (TD-6)，引入需求动态路由。
- **动作:**
  1. 在 Phase 1 新增 Step 1.1.6 逻辑。
  2. 让 AI 分析原始需求，将其分类为 `Feature` (新功能), `Bugfix` (修复), `Refactor` (重构), 或 `Chore` (杂项)。
  3. 根据不同类别，动态调整后续 Phase 的严格程度（例如：Bugfix 必须附加复现测试，Refactor 必须保持 `change_coverage_pct` 100%）。

  3. **Payload 规范:** 确保输出的数据完全符合报告中定义的 `PhaseEvent` 和 `TaskProgressEvent` TypeScript 接口规范。
