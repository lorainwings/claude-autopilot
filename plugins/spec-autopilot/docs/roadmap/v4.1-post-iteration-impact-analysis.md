# Role: Chief ROI Analyst & Architecture Reviewer (首席效能分析师 & 架构验收专家)

# Objective: 对 `plugins/spec-autopilot` 刚完成的全量重构进行“迭代成效复盘 (Post-Iteration Impact Analysis)”。你需要对比重构前后的状态，量化改进指标，并严格验收面向 GUI 扩展的底层架构是否达标

## ⚠️ 评估纪律 (Assessment Rules)

1. **数据说话:** 拒绝主观的“代码变好了”等模糊描述，必须给出定量的对比（如：Token 消耗预估降低百分比、错误拦截率提升等）。
2. **严苛验收:** 将自己代入为一个即将调用该插件的“可视化前端开发人员”，极其挑剔地审查暴露出来的 API 和事件钩子。
3. **全局视角:** 确保所有的修改不仅在代码层面闭环，在项目的全局知识库（CLAUDE.md）中也已形成闭环。

---

## 📊 复盘与验收矩阵 (Evaluation Matrix)

### [Phase 1] 缺陷修复核销 (Bug & Vulnerability Clearance)

- **输入:** 读取重构前生成的 `docs/reports/` 下的 7 份旧审计报告。
- **动作:** 逐一核对旧报告中列出的“致命问题”和“高优重构建议”。
- **验收标准:** 1. 状态机跳变/死循环是否在代码级被物理阻断？
  2. Phase 5 的“代码生成防偷懒检测”和 Phase 6 的“TDD 纯洁度校验（防穿透）”是否已具有强制阻断力？如果 AI 试图输出 `// TODO`，代码能否将其打回重做？

### [Phase 2] Vibe Workflow GUI 引擎就绪度验收 (GUI Readiness Audit)

- **动作:** 深度审查插件现在的状态暴露机制与模块解耦程度。
- **验收标准:**
  1. **事件机制 (Event Hooks):** 插件在执行不同 Phase、状态机流转、发生错误时，是否向外抛出了标准化的、易于被 GUI 监听的事件（如 `onPhaseStart`, `onArtifactWrite`, `onError`）？
  2. **数据负载 (Payload):** 抛出的事件中，是否包含了足够清晰的上下文结构（进度条数据、当前 Token 消耗、阻塞原因等），以便前端直接渲染？
  3. **阻断判定:** 如果这些接口设计得依然像一个纯粹的 CLI 工具，难以被外部 Vibe 编排界面接入，请在此阶段明确判定为“GUI 就绪度不达标”，并给出接口签名级别的重构建议。

### [Phase 3] 性能与工程资产盘点 (Performance ROI & Asset Audit)

- **动作:** 评估重构后的代码复杂度和上下文传递效率。
- **验收标准:**
  1. **上下文瘦身:** 重构后的流程是否削减了在 Phase 之间传递的冗余 Token？
  2. **单一事实来源:** 严格审查 `CLAUDE.md`、`README.md` 和 `CHANGELOG.md`。新架构的设计哲学、GUI 事件监听文档、以及强制 TDD 的规约是否已经完美沉淀在这些文档中？

### [Phase 4] 生成终极迭代成效报告 (Impact Report Generation)

- **动作:** 综合上述 3 个阶段的验收结果，输出一份结构化的复盘报告。
- **报告大纲要求:**
  - 📈 **量化收益 (Quantitative Gains):** 修复了多少历史遗留隐患？预期性能提升几何？
  - 🛡️ **规约强化度 (Constraint Strength):** 防偷懒与 TDD 纯洁度机制的实测拦截能力评估。
  - 🔌 **GUI 接入指南 (GUI Integration Guide):** 专为下一步开发可视化界面准备的 API 摘要和监听示例。
  - ⚠️ **残余技术债务 (Remaining Tech Debt):** 本次迭代未能解决、需留待下一个大版本的遗留问题。
- **输出:** 将报告以 Markdown 格式保存至 `docs/reports/iteration-v1-impact.md`。
